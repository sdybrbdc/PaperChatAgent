## 第一章：FastAPI

本章聚焦 FastAPI 本身，包括应用创建、路由、参数接收、Pydantic、依赖注入、生命周期、Swagger/OpenAPI 以及开发中最常见的一套组合。对于当前项目来说，FastAPI 负责对外暴露 API、接收前端请求，并通过 SSE 把后端任务状态持续推给前端。

### FastAPI、ASGI、Uvicorn 分别是什么

很多人刚接触 FastAPI 时，会看到下面这段代码：

```python
import uvicorn
from fastapi import FastAPI

app = FastAPI()

if __name__ == "__main__":
    uvicorn.run(app, host="0.0.0.0", port=8000)
```

这里面有三个容易混在一起的角色：

- `FastAPI`：Web 框架，负责定义路由、参数校验、依赖注入、自动文档等
- `ASGI`：Python Web 应用的异步接口标准
- `uvicorn`：ASGI 服务器，负责真正监听端口、接收请求、返回响应

可以这样理解：

- `FastAPI` 负责“你要提供什么接口”
- `uvicorn` 负责“把这个接口服务跑起来”

也就是说，`app = FastAPI()` 只是创建了一个应用对象，并没有真正启动 HTTP 服务。  
要让浏览器或前端能访问它，还需要一个服务器进程，`uvicorn` 就是这个角色。

---

### 为什么要通过 `uvicorn` 启动

因为 FastAPI 本身不是一个独立的 Web 服务器，它只是一个符合 ASGI 规范的应用。

所以这两者分工是：

- FastAPI：定义业务逻辑
- Uvicorn：启动服务并处理网络通信

例如：

```python
import uvicorn
from fastapi import FastAPI

app = FastAPI()

@app.get("/ping")
def ping():
    return {"msg": "pong"}

if __name__ == "__main__":
    uvicorn.run(app, host="0.0.0.0", port=8000)
```

启动后你才能访问：

- `http://localhost:8000/ping`
- `http://localhost:8000/docs`

`uvicorn` 特别适合 FastAPI，因为它支持：

- `async/await`
- WebSocket
- 高并发
- 流式响应
- SSE

你的当前项目里就有 SSE 接口，这也是 FastAPI + Uvicorn 很常见的一个组合。

---

### 常见启动方式

#### 在代码里启动

```python
if __name__ == "__main__":
    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=8000)
```

优点：

- 简单直接
- 单文件 demo 很方便

#### 在命令行启动

```bash
uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

含义：

- `main`：Python 文件名 `main.py`
- `app`：文件里的 FastAPI 应用实例
- `--reload`：代码修改后自动重启

这种方式在开发阶段更常见。

#### 开发和生产的一般区别

- 开发环境：常用 `uvicorn --reload`
- 生产环境：通常不会直接用 `--reload`，而是配合进程管理或多 worker 部署

### `app = FastAPI()` 是做什么的

`FastAPI()` 用来创建一个 FastAPI 应用实例。后续的大部分配置、路由注册、中间件挂载，都是基于这个 `app` 来完成的。

常见写法：

```python
from fastapi import FastAPI

app = FastAPI(
    title="My API",
    description="接口说明文档",
    version="1.0.0",
    docs_url="/docs",
    redoc_url="/redoc",
    openapi_url="/openapi.json",
)
```

常见参数：

- `title`：接口文档标题
- `description`：接口描述
- `version`：版本号
- `docs_url`：Swagger UI 文档地址
- `redoc_url`：ReDoc 文档地址
- `openapi_url`：OpenAPI Schema 地址
- `lifespan`：应用启动和关闭时的生命周期逻辑
- `openapi_tags`：给文档页面设置接口分组说明

---

### `app.include_router()` 是做什么的

`include_router()` 用于把其他模块中定义好的路由注册到主应用中。

它的主要作用是让项目结构更清晰，适合把不同业务拆分到不同文件中，例如：

- 用户相关接口放到 `user.py`
- 文章相关接口放到 `article.py`
- 管理后台接口放到 `admin.py`

示例：

```python
from fastapi import FastAPI
from app.routers.user import router as user_router

app = FastAPI()

app.include_router(user_router, prefix="/users", tags=["users"])
```

可以这样理解：

- `router`：一组已经定义好的接口
- `prefix="/users"`：给这组接口统一加前缀
- `tags=["users"]`：在文档中按标签分类展示

如果路由文件里有：

```python
from fastapi import APIRouter

router = APIRouter()

@router.get("/")
def get_users():
    return ["Tom", "Jerry"]
```

那么最终接口路径就是：

```text
GET /users/
```

---

### `APIRouter()` 是做什么的

`APIRouter()` 是路由分组工具。一般项目不会把所有接口都写在 `main.py` 里，而是把相关接口放到同一个 router 里。

示例：

```python
from fastapi import APIRouter

router = APIRouter(prefix="/items", tags=["items"])

@router.get("/")
def list_items():
    return [{"id": 1, "name": "book"}]
```

然后在主应用中注册：

```python
app.include_router(router)
```

适合这样拆分：

- `user_router`
- `order_router`
- `knowledge_router`

---

### `app.add_middleware()` 是做什么的

`add_middleware()` 用于给应用添加中间件。

中间件会在请求到达具体接口之前，以及响应返回客户端之前执行。你可以把它理解成请求处理过程中的一层“统一拦截器”。

常见用途：

- 处理跨域 CORS
- 打印请求日志
- 统计接口耗时
- 统一鉴权
- 给请求注入公共上下文

示例：

```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

这个例子表示给应用增加了 CORS 中间件，允许前端跨域访问该服务。

### 常用路由 API

最常用的是这些装饰器：

- `@app.get()`
- `@app.post()`
- `@app.put()`
- `@app.delete()`
- `@app.patch()`

它们用于定义不同 HTTP 方法的接口。

示例：

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/ping")
def ping():
    return {"msg": "pong"}

@app.post("/items")
def create_item():
    return {"msg": "item created"}
```

含义：

- `GET`：查询数据
- `POST`：新建数据
- `PUT`：整体更新数据
- `PATCH`：部分更新数据
- `DELETE`：删除数据

这些装饰器也常见于 `router` 上：

```python
@router.get("/")
@router.post("/")
```

### 请求参数怎么接收

FastAPI 很强的一点，是它能直接根据函数签名解析参数来源。

#### 路径参数

```python
@app.get("/users/{user_id}")
def get_user(user_id: int):
    return {"user_id": user_id}
```

这里的 `user_id` 来自 URL 路径。

#### 查询参数

```python
@app.get("/search")
def search(keyword: str, page: int = 1):
    return {"keyword": keyword, "page": page}
```

请求示例：

```text
GET /search?keyword=fastapi&page=2
```

#### `Query()`

用于给查询参数补充默认值、校验规则和文档说明。

```python
from fastapi import Query

@app.get("/items")
def list_items(page: int = Query(1, ge=1), size: int = Query(10, le=100)):
    return {"page": page, "size": size}
```

#### `Path()`

用于给路径参数增加校验和说明。

```python
from fastapi import Path

@app.get("/items/{item_id}")
def get_item(item_id: int = Path(..., ge=1)):
    return {"item_id": item_id}
```

#### `Body()`

用于显式声明请求体参数。

```python
from fastapi import Body

@app.post("/login")
def login(username: str = Body(...), password: str = Body(...)):
    return {"username": username}
```

### 为什么推荐用 Pydantic 模型接收请求体

在真实项目里，更推荐用 Pydantic 模型定义请求体，而不是写一堆零散参数。

示例：

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class ItemCreate(BaseModel):
    name: str
    price: float
    in_stock: bool = True

@app.post("/items")
def create_item(item: ItemCreate):
    return {"item": item.model_dump()}
```

好处：

- 自动校验参数类型
- 自动生成 Swagger 文档
- 字段定义清晰
- 更适合复杂请求结构

### 依赖注入 `Depends()`

`Depends()` 是 FastAPI 很核心的功能，用来复用公共逻辑。

常见用途：

- 登录校验
- 获取数据库连接
- 提取当前用户
- 参数预处理

示例：

```python
from fastapi import Depends, FastAPI

app = FastAPI()

def get_current_user():
    return {"id": 1, "name": "Tom"}

@app.get("/me")
def read_me(user=Depends(get_current_user)):
    return user
```

你可以把 `Depends()` 理解成：

- 先执行依赖函数
- 再把结果注入到当前接口

### 异常处理

#### `HTTPException`

用于主动返回 HTTP 错误。

```python
from fastapi import HTTPException

@app.get("/users/{user_id}")
def get_user(user_id: int):
    if user_id != 1:
        raise HTTPException(status_code=404, detail="User not found")
    return {"user_id": user_id}
```

#### `app.exception_handler()`

用于注册自定义异常处理器。

```python
from fastapi import Request
from fastapi.responses import JSONResponse

@app.exception_handler(ValueError)
async def value_error_handler(request: Request, exc: ValueError):
    return JSONResponse(
        status_code=400,
        content={"detail": str(exc)}
    )
```

作用：

- 捕获指定异常
- 返回统一格式的错误响应

### 生命周期 `lifespan`

用于管理应用启动和关闭时的生命周期逻辑。

现在更推荐使用 `lifespan`，相较于旧写法 `@app.on_event("startup")` / `@app.on_event("shutdown")` 更现代。

示例：

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    print("startup")
    yield
    print("shutdown")

app = FastAPI(lifespan=lifespan)
```

常见用途：

- 初始化数据库连接
- 加载机器学习模型
- 初始化缓存
- 服务关闭时释放资源

### 开发中最常见的一套组合

实际项目里，最常见的一套组合通常是：

这样项目结构通常会更清晰，也更容易扩展。

## 第二章：LangGraph

本章聚焦 LangGraph 的状态图编排能力，包括 `State`、`Node`、`Edge`、条件路由、循环、异步执行以及它在当前项目中如何承接“搜索 -> 阅读 -> 分析 -> 写作 -> 汇总”的主流程。对于当前项目来说，LangGraph 负责定义业务流程骨架，而不是具体的智能行为本身。

### LangGraph 构建与使用知识整理

本节整理当前项目里的 LangGraph 使用方式，以及官方文档中的核心概念、节点构造、边构造、条件判断、循环、异步调用和 MCP 集成方式。

参考官方文档：

- [LangGraph Graph API](https://docs.langchain.com/oss/python/langgraph/graph-api)
- [LangGraph Use Graph API](https://docs.langchain.com/oss/python/langgraph/use-graph-api)
- [LangChain MCP](https://docs.langchain.com/oss/python/langchain/mcp)

#### LangGraph 是什么

LangGraph 可以理解成一个“可控流程编排框架”。

它不是单纯的 LLM 调用工具，而是用“图”的方式组织复杂任务流程。

在普通代码里，我们可能这样写：

```python
result1 = await search()
result2 = await read(result1)
result3 = await analyse(result2)
result4 = await write(result3)
```

LangGraph 把这些步骤抽象成：

- `State`：流程共享状态
- `Node`：处理状态的节点函数
- `Edge`：节点之间的流转关系
- `Conditional Edge`：根据状态决定下一步去哪
- `START` / `END`：图的开始和结束

所以 LangGraph 更像是：

```text
START
  |
search_node
  |
reading_node
  |
analyse_node
  |
writing_node
  |
report_node
  |
END
```

当前项目的论文研究流程就是这种模式。

#### LangGraph 的官方核心三件套

##### State：共享状态

`State` 是整个图执行过程中的共享数据结构。

每个节点都会接收当前 `state`，并返回一部分状态更新。

常见写法是 `TypedDict`：

```python
from typing import TypedDict

class ResearchState(TypedDict):
    query: str
    papers: list[str]
    summary: str
```

也可以使用：

- `TypedDict`
- `dataclass`
- `Pydantic BaseModel`

官方更常见的是 `TypedDict`，因为它简单、性能也更好。Pydantic 更适合输入输出需要严格校验的场景。

##### Node：节点函数

`Node` 是图里的执行单元，本质上就是一个普通函数或异步函数。

```python
async def search_node(state: ResearchState) -> dict:
    papers = await search_papers(state["query"])
    return {"papers": papers}
```

节点通常应该返回“状态增量”，而不是完整状态。

例如：

```python
return {"papers": papers}
```

不建议每个节点都返回整个 state：

```python
return state
```

因为这样容易覆盖其他节点产生的数据，尤其在并发节点或多分支场景下更容易出问题。

##### Edge：边

`Edge` 决定节点执行顺序。

```python
builder.add_edge("search_node", "reading_node")
```

这表示：

```text
search_node 执行完后，进入 reading_node
```

#### 当前项目里的 LangGraph 在哪里

当前项目核心流程主要在：

- `src/agents/orchestrator.py`
- `src/models/state_models.py`

#### 最小 LangGraph 构建模板

一个最小 LangGraph 通常是这样：

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class MyState(TypedDict):
    query: str
    result: str

async def search_node(state: MyState) -> dict:
    return {"result": f"search result for {state['query']}"}

builder = StateGraph(MyState)

builder.add_node("search_node", search_node)
builder.add_edge(START, "search_node")
builder.add_edge("search_node", END)

graph = builder.compile()

result = await graph.ainvoke({"query": "LangGraph 是什么"})
```

关键顺序是：

```text
定义 State
  -> 定义 Node
  -> 创建 StateGraph
  -> add_node
  -> add_edge / add_conditional_edges
  -> compile
  -> invoke / ainvoke
```

#### `StateGraph(...)` 是什么

`StateGraph` 是 LangGraph 用来构建图的对象。

```python
builder = StateGraph(MyState)
```

这里的 `MyState` 是状态结构定义。

LangGraph 会根据这个状态定义来判断：

- 图执行时有哪些字段
- 节点返回的数据如何合并到 state
- 某些字段是否有 reducer
- 输入输出结构是否符合预期

当前项目中，状态相关结构主要在 `src/models/state_models.py`。

#### `builder.set_entry_point("search_node")` 是什么

`builder.set_entry_point("search_node")` 的意思是设置图的入口节点。

它等价于告诉 LangGraph：

```text
图开始执行后，第一个进入 search_node
```

在新版官方示例里，也常见这种写法：

```python
builder.add_edge(START, "search_node")
```

两种写法表达的语义类似：

```python
builder.set_entry_point("search_node")
```

等价理解为：

```python
builder.add_edge(START, "search_node")
```

如果当前项目里看到：

```python
builder.set_entry_point("search_node")
```

就说明当前论文研究流程是从 `search_node` 开始的。

#### `add_node` 节点构造

当前环境中 `StateGraph.add_node` 的核心签名可以理解为：

```python
builder.add_node(
    node,
    action=None,
    *,
    defer=False,
    metadata=None,
    input_schema=None,
    retry_policy=None,
    cache_policy=None,
    destinations=None,
)
```

常用写法一：

```python
builder.add_node("search_node", search_node)
```

常用写法二：

```python
builder.add_node(search_node)
```

如果只传函数，LangGraph 会尝试使用函数名作为节点名。

参数说明：

- `node`：节点名，或者节点函数本身
- `action`：节点具体执行函数
- `defer`：是否延迟执行，常用于需要在最后阶段执行的节点
- `metadata`：节点元数据，常用于调试、可视化、追踪
- `input_schema`：指定节点输入 schema
- `retry_policy`：节点失败后的重试策略
- `cache_policy`：节点缓存策略
- `destinations`：声明可能去向，常用于图可视化或命令式路由

节点函数可以是同步函数：

```python
def node_a(state: MyState) -> dict:
    return {"result": "ok"}
```

也可以是异步函数：

```python
async def node_a(state: MyState) -> dict:
    result = await call_llm()
    return {"result": result}
```

还可以接收额外的运行配置：

```python
from langchain_core.runnables import RunnableConfig

async def node_a(state: MyState, config: RunnableConfig) -> dict:
    user_id = config.get("configurable", {}).get("user_id")
    return {"result": user_id}
```

新版本 LangGraph 也支持 runtime/context 风格，用来传递运行期依赖。

#### `add_edge` 固定边构造

`add_edge` 用于构造固定流程。

签名可以理解为：

```python
builder.add_edge(start_key, end_key)
```

示例：

```python
builder.add_edge("search_node", "reading_node")
builder.add_edge("reading_node", "analyse_node")
builder.add_edge("analyse_node", "writing_node")
builder.add_edge("writing_node", END)
```

表示：

```text
search_node -> reading_node -> analyse_node -> writing_node -> END
```

如果一个节点永远固定进入下一个节点，就用 `add_edge`。

#### `add_conditional_edges` 条件边构造

`add_conditional_edges` 用于“根据状态决定下一步去哪”。

签名可以理解为：

```python
builder.add_conditional_edges(
    source,
    path,
    path_map=None,
)
```

参数说明：

- `source`：从哪个节点执行完后开始判断
- `path`：路由函数，根据 state 返回下一步
- `path_map`：可选映射，把路由函数返回值映射成真实节点名

最简单写法：

```python
def should_continue(state: MyState) -> str:
    if state["need_more"]:
        return "search_node"
    return END

builder.add_conditional_edges("analyse_node", should_continue)
```

这表示：

```text
analyse_node 执行完
  -> 如果 need_more=True，回到 search_node
  -> 否则结束
```

如果希望路由函数返回更抽象的名字，可以使用 `path_map`：

```python
def route(state: MyState) -> str:
    if state["error"]:
        return "error"
    if state["done"]:
        return "finish"
    return "continue"

builder.add_conditional_edges(
    "analyse_node",
    route,
    {
        "error": "handle_error_node",
        "finish": END,
        "continue": "writing_node",
    },
)
```

这样路由函数不用直接关心真实节点名，代码更清晰。

#### 条件函数应该怎么写

条件函数本质上就是普通函数。

它通常接收当前 state，返回：

- 下一个节点名
- `END`
- 多个节点名
- `Command`
- `Send`

最常见的是返回字符串：

```python
def route_after_search(state: MyState) -> str:
    if not state["papers"]:
        return "handle_error_node"
    return "reading_node"
```

也可以返回多个节点，实现并行分支：

```python
def route_to_parallel_nodes(state: MyState) -> list[str]:
    return ["analyse_node", "summary_node"]
```

多个节点在同一个 superstep 中执行，然后它们返回的状态会按照 reducer 合并。

#### 如何实现循环

LangGraph 的循环不是 `while True` 写在一个节点里，而是通过条件边把流程路由回前面的节点。

例如“写作 -> 审核 -> 不通过继续写 -> 通过结束”：

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class WriteState(TypedDict):
    draft: str
    score: int
    revision_count: int

async def write_node(state: WriteState) -> dict:
    draft = await generate_draft(state.get("draft", ""))
    return {
        "draft": draft,
        "revision_count": state.get("revision_count", 0) + 1,
    }

async def review_node(state: WriteState) -> dict:
    score = await review_draft(state["draft"])
    return {"score": score}

def should_rewrite(state: WriteState) -> str:
    if state["score"] >= 80:
        return END
    if state["revision_count"] >= 3:
        return END
    return "write_node"

builder = StateGraph(WriteState)
builder.add_node("write_node", write_node)
builder.add_node("review_node", review_node)

builder.add_edge(START, "write_node")
builder.add_edge("write_node", "review_node")
builder.add_conditional_edges("review_node", should_rewrite)

graph = builder.compile()
```

流程是：

```text
START -> write_node -> review_node
                       |
                       | score < 80 and revision_count < 3
                       v
                    write_node

如果分数达标或超过最大次数 -> END
```

实现循环时必须有明确退出条件，否则容易无限执行。

#### 当前项目中的条件判断

当前项目里的 `orchestrator.py` 使用了条件路由函数，比如 `condition_handler`。

它的作用大致是：

```text
检查当前 state
  -> 如果当前节点有错误，进入 handle_error_node
  -> 如果当前步骤完成，进入下一个节点
  -> 如果已经结束，进入 END
```

也就是说，当前项目不是简单地写死：

```text
search_node -> reading_node -> analyse_node -> writing_node -> report_node
```

而是在每个阶段结束后检查状态：

- `search_node_error`
- `reading_node_error`
- `analyse_node_error`
- `writing_node_error`
- `report_node_error`
- `current_step`

如果发现错误，就可以统一进入错误处理节点。

这类设计的好处是：

- 每个节点只关心自己的任务
- 错误判断集中到路由函数里
- 前端可以通过状态流知道当前执行到哪一步
- 后续要新增节点时，可以只调整路由逻辑

#### `compile()` 是什么

构建完节点和边以后，需要调用：

```python
graph = builder.compile()
```

`compile()` 的作用是把“图定义”编译成“可执行图”。

在 `compile()` 之前，`builder` 只是一个图构造器。

在 `compile()` 之后，`graph` 才能执行：

```python
graph.invoke(...)
await graph.ainvoke(...)
graph.stream(...)
graph.astream(...)
```

`compile()` 常见参数包括：

- `checkpointer`：检查点，用于保存中间状态，支持恢复、记忆、多轮 thread
- `cache`：缓存节点结果
- `store`：跨 thread 的持久化存储
- `interrupt_before`：在某些节点前中断，做人类确认
- `interrupt_after`：在某些节点后中断
- `debug`：调试模式
- `name`：图名称

当前项目如果未来要支持“任务中断后恢复”，优先研究 `checkpointer`。

#### `ainvoke()` 是调用 LLM 吗

不是。

```python
await self.graph.ainvoke(...)
```

这句话的意思是“异步执行整个 LangGraph 图”。

它不等价于直接调用 LLM。

真正是否调用 LLM，取决于图里的节点函数做了什么。

例如：

```python
async def search_node(state):
    return await search_agent.run(...)
```

如果 `search_agent.run(...)` 内部调用了 AutoGen Agent 或模型客户端，那这个节点里就会调用 LLM。

所以可以这样理解：

```text
graph.ainvoke()
  -> 执行图
  -> 图进入 search_node
  -> search_node 内部可能调用 LLM
  -> 图进入 reading_node
  -> reading_node 内部可能调用 LLM
  -> ...
```

当前项目中：

```python
await self.graph.ainvoke({
    "state_queue": self.state_queue,
    "value": initial_state,
})
```

这里传入的是图的初始状态。

其中：

- `state_queue`：异步队列，用于把执行状态推给前端
- `value`：当前论文研究流程真正使用的业务状态

#### `state_queue.put(...)` 是什么

当前项目里有类似代码：

```python
await self.state_queue.put(
    BackToFrontData(
        step=ExecutionState.FINISHED,
        state="finished",
        data=None,
    )
)
```

这不是 LangGraph 自带能力，而是项目自己用 `asyncio.Queue` 做的前后端通信机制。

含义是：

```text
把一条状态消息放进异步队列
```

前端通过 SSE 监听服务端事件。

后端事件生成器不断从 `state_queue` 里取消息，然后推给前端。

#### `BackToFrontData(... FINISHED ...)` 是标准结束吗

对当前项目来说，它是“项目自定义的标准结束事件”。

```python
BackToFrontData(
    step=ExecutionState.FINISHED,
    state="finished",
    data=None,
)
```

可以理解成告诉前端：

```text
整个任务流程已经完成
```

它不是 LangGraph 官方规定的结束格式。

LangGraph 官方结束是通过 `END` 节点表达的。

当前项目有两层结束概念：

- LangGraph 层：流程走到 `END`
- 前端通信层：向 `state_queue` 放入 `ExecutionState.FINISHED`

所以这段代码负责的是“通知前端任务完成”。

#### Reducer：状态如何合并

LangGraph 中，每个节点返回的是状态更新。

默认情况下，同名字段会被覆盖。

例如：

```python
class MyState(TypedDict):
    result: str

def node_a(state: MyState):
    return {"result": "A"}

def node_b(state: MyState):
    return {"result": "B"}
```

如果两个节点都写 `result`，后面的值可能覆盖前面的值。

如果想追加列表，需要 reducer。

示例：

```python
from typing import Annotated, TypedDict
import operator

class MyState(TypedDict):
    logs: Annotated[list[str], operator.add]
```

这样多个节点返回：

```python
return {"logs": ["search finished"]}
```

最终会追加到 `logs`，而不是覆盖。

聊天消息场景常用：

```python
from langgraph.graph.message import add_messages
from typing import Annotated, TypedDict

class ChatState(TypedDict):
    messages: Annotated[list, add_messages]
```

`add_messages` 会按消息 ID 合并消息，适合对话历史。

#### `Command`：更新状态并决定下一步

有些节点不仅要返回状态，还想直接决定下一步去哪。

这时可以使用 `Command`。

示例：

```python
from typing import Literal
from langgraph.types import Command

def node_a(state) -> Command[Literal["node_b", "node_c"]]:
    if state["ok"]:
        return Command(
            update={"status": "ok"},
            goto="node_b",
        )

    return Command(
        update={"status": "failed"},
        goto="node_c",
    )
```

`Command` 适合：

- 节点内部已经知道下一步
- 想减少额外条件边函数
- 状态更新和路由强相关

但如果流程比较复杂，建议仍然使用 `add_conditional_edges`，可读性更好。

#### `Send`：动态并行和 Map-Reduce

`Send` 适合动态创建多个并行任务。

例如有多篇论文，需要分别分析：

```python
from langgraph.constants import Send

def continue_to_paper_analysis(state):
    return [
        Send("analyse_one_paper_node", {"paper": paper})
        for paper in state["papers"]
    ]
```

然后：

```python
builder.add_conditional_edges(
    "reading_node",
    continue_to_paper_analysis,
)
```

这表示：

```text
reading_node 结束后
  -> 根据 papers 数量动态创建多个 analyse_one_paper_node
  -> 每个节点处理一篇论文
  -> 最后通过 reducer 合并结果
```

如果当前项目后续要优化多篇论文分析速度，可以考虑这种模式。

#### LangGraph 调用 MCP 是什么

MCP 是 Model Context Protocol，用来让模型或 Agent 连接外部工具、资源和上下文。

LangGraph 本身负责流程编排。

MCP 负责提供外部工具。

两者组合后就是：

```text
LangGraph 决定什么时候调用工具
MCP 提供可以调用的工具列表
Agent 或 Node 具体执行工具调用
```

例如：

```text
search_node
  -> 从 MCP server 获取 search tool
  -> 调用 search tool
  -> 把结果写入 state
```



#### 在 LangGraph 节点里调用 MCP

一种直接方式是在节点里加载 MCP tools，然后调用工具。

示例：

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langchain_mcp_adapters.client import MultiServerMCPClient

class ResearchState(TypedDict):
    query: str
    search_result: str

async def mcp_search_node(state: ResearchState) -> dict:
    client = MultiServerMCPClient(
        {
            "search": {
                "url": "https://example.com/mcp",
                "transport": "http",
            }
        }
    )

    tools = await client.get_tools()
    search_tool = next(tool for tool in tools if tool.name == "search")

    result = await search_tool.ainvoke({"query": state["query"]})

    return {"search_result": str(result)}

builder = StateGraph(ResearchState)
builder.add_node("mcp_search_node", mcp_search_node)
builder.add_edge(START, "mcp_search_node")
builder.add_edge("mcp_search_node", END)

graph = builder.compile()
```

这种方式适合：

- 明确知道要调用哪个工具
- 工具调用逻辑比较固定
- 不需要让 LLM 自己选择工具

#### 在 Agent 里使用 MCP tools

另一种方式是把 MCP tools 交给 Agent。

示例：

```python
from langchain_mcp_adapters.client import MultiServerMCPClient
from langgraph.prebuilt import create_react_agent

client = MultiServerMCPClient(
    {
        "tools": {
            "url": "https://example.com/mcp",
            "transport": "http",
        }
    }
)

tools = await client.get_tools()

agent = create_react_agent(
    model,
    tools,
)

result = await agent.ainvoke(
    {
        "messages": [
            {"role": "user", "content": "帮我搜索 LangGraph 的条件边用法"}
        ]
    }
)
```

这种方式适合：

- 工具有多个
- 需要模型自己判断是否调用工具
- 需要模型决定调用哪个工具
- 需要多轮 tool calling

#### 当前项目如果要接 MCP，应该放在哪里


如果 MCP 是用来增强检索，可以放在：

其中 `mcp_retrieval_node` 专门负责：

- 写入 state

#### 当前项目和 MCP 的关系

从当前项目代码看，已有的核心调用链主要是：

```text
FastAPI API
  -> PaperAgentOrchestrator
  -> LangGraph
  -> search / reading / analyse / writing / report agents
  -> AutoGen / model client
  -> 自定义大模型接口
```

MCP 暂时不是主链路必需组件。

如果加入 MCP，它更像是增强工具层：

```text
FastAPI API
  -> PaperAgentOrchestrator
  -> LangGraph
  -> MCP tool node
  -> AutoGen / LLM
```

所以 MCP 不应该替代 LangGraph。

更准确地说：

- LangGraph 管流程
- AutoGen 管多 Agent 协作或 Agent 行为
- Model Client 管模型连接
- MCP 管外部工具和上下文接入
- SSE 管后端到前端的实时状态推送

#### 常见坑

##### 坑一：以为 `ainvoke()` 就是调用 LLM

`ainvoke()` 只是执行图。

是否调用 LLM，取决于节点内部。

##### 坑二：节点直接修改 state

不推荐这样：

```python
state["result"] = "xxx"
return state
```

更推荐：

```python
return {"result": "xxx"}
```

这样 LangGraph 能更明确地管理状态合并。

##### 坑三：循环没有退出条件

条件边回到前面节点时，必须有明确终止条件。

例如：

```python
if retry_count >= 3:
    return END
```

否则可能无限循环。

##### 坑四：并行节点写同一个字段但没有 reducer

多个节点同时返回同一个字段时，默认是覆盖，不是追加。

需要追加列表时，应使用：

```python
Annotated[list[str], operator.add]
```

##### 坑五：把复杂对象直接塞进 state，导致序列化问题

如果 state 里放了模型客户端、数据库连接、异步队列、文件句柄等对象，后续做日志、序列化、持久化、checkpoint 时容易出问题。

当前项目把 `state_queue` 放进 state 是为了 SSE 通信，能工作，但如果未来使用 `checkpointer`，要特别注意它可能不能被正常序列化。

##### 坑六：多个请求共享同一个 queue 或 orchestrator

如果 Web API 每次请求没有隔离自己的 orchestrator 和 queue，可能出现状态串流混乱。

比较安全的做法是：

```text
每个请求创建自己的 orchestrator
每个 orchestrator 持有自己的 state_queue
每个 SSE 连接只消费自己的 queue
```

##### 坑七：异步任务被取消但没有正确清理

之前看到过类似：

```text
Task was destroyed but it is pending
aclose(): asynchronous generator is already running
```

这类问题通常和以下情况有关：

- 前端断开 SSE 连接
- 后端异步 generator 没有正确关闭
- LangGraph 任务还在运行
- queue 消费者退出了，但生产者还在 put

解决方向通常是：

- 捕获 `asyncio.CancelledError`
- 在 SSE 断开时取消后台任务
- 避免多个协程同时关闭同一个 async generator
- 给 LangGraph 任务添加统一生命周期管理

#### 当前项目推荐学习顺序

建议按这个顺序学习：

```text
1. 先看 src/models/state_models.py
2. 再看 src/agents/orchestrator.py 的 build_graph
3. 理解 search_node -> reading_node -> analyse_node -> writing_node -> report_node
4. 理解 condition_handler 如何决定下一步
5. 理解 state_queue 如何把状态推给 SSE
6. 再看每个 agent 内部如何调用 AutoGen 和模型
7. 最后再考虑 MCP、checkpoint、Command、Send 等高级能力
```

不要一开始就从 MCP 或复杂 Agent 入手。

当前项目最核心的主线是：

```text
FastAPI 接口
  -> 创建 orchestrator
  -> LangGraph 执行节点
  -> 节点调用 Agent
  -> Agent 调用模型
  -> 状态通过 queue 推给 SSE
  -> 前端实时展示
```

#### 一句话总结

LangGraph 在当前项目里负责“把论文研究任务拆成多个可控节点并按条件流转”，`builder.set_entry_point("search_node")` 表示从检索节点开始，`add_node` 注册处理逻辑，`add_edge` 注册固定流程，`add_conditional_edges` 注册条件分支和循环，`graph.ainvoke()` 执行整个图，而 `state_queue.put(...)` 是项目自己用来把执行状态通过 SSE 推给前端的通信机制；MCP 后续可以作为工具层接入某些节点，但不替代 LangGraph 的流程编排职责。

#### `ModelInfo` 到底是什么

`ModelInfo` 来自：

```python
from autogen_core.models import ModelInfo
```

官方文档里它是一个 `TypedDict`，表示“模型能力描述”。

它不是账号配置，也不是推理参数，而是告诉 AutoGen：

- 这个模型能不能看图
- 能不能函数调用
- 能不能 JSON 输出
- 属于哪个模型家族
- 能不能结构化输出
- 能不能支持多个 system message

#### `ModelInfo` 的字段有哪些

官方 stable 文档和本地 `0.7.5` 注解一致，字段如下：

- `vision: bool`
- `function_calling: bool`
- `json_output: bool`
- `family: ModelFamily | str`
- `structured_output: bool`
- `multiple_system_messages: bool | None`

字段含义可以这样理解：

##### `vision`

- `True`：模型支持图片输入
- `False`：只支持纯文本

##### `function_calling`

- `True`：模型支持 tools / function calling
- `False`：不能稳定调用工具

##### `json_output`

- `True`：支持 JSON 模式输出
- `False`：不能可靠使用 JSON mode

注意：

`json_output` 不等于 `structured_output`。

##### `family`

这是模型家族标识。

官方文档里推荐使用 `ModelFamily` 里的常量，当前 stable 文档列出的家族包括：

- `gpt-5`
- `gpt-41`
- `gpt-45`
- `gpt-4o`
- `o1`
- `o3`
- `o4`
- `gpt-4`
- `gpt-35`
- `r1`
- `gemini-*`
- `claude-*`
- `llama-*`
- `mistral`
- `ministral`
- `pixtral`
- `unknown`
- `any`

如果模型家族不在官方枚举里，也可以传字符串。

##### `structured_output`

- `True`：支持结构化输出
- `False`：不适合用 Pydantic schema 做结构化生成

##### `multiple_system_messages`

- `True`：支持多个非连续 system message
- `False`：不支持
- `None`：未知或不特别声明

#### 什么时候必须传 `ModelInfo`

官方文档明确写了：

- 如果 `model` 不是 AutoGen 已知的有效 OpenAI 模型名，`model_info` 就应当提供

这点对你项目很关键。

因为你现在常见的模型名像：

- `deepseek-ai/DeepSeek-V3.2`
- `Qwen/Qwen3-32B`
- `Qwen/Qwen3-Embedding-8B`

这些都不是 OpenAI 官方标准模型名。  
所以你现在手动构造 `ModelInfo(...)` 这一步是合理的，而且对于 OpenAI-compatible 第三方模型来说通常是必要的。

#### 当前项目里 `ModelInfo(...)` 这段代码是什么意思

你现在的代码本质上是在给 AutoGen 做能力声明：

```python
model_info = ModelInfo(
    vision=vision,
    function_calling=function_calling,
    json_output=json_output,
    family=family,
    structured_output=structured_output
)
```

这段的作用不是“传模型参数”，而是：

- 帮 AutoGen 判断能不能走工具调用
- 帮 AutoGen 判断能不能走 JSON 输出
- 帮 AutoGen 判断能不能走结构化输出
- 在模型不是官方已知名字时补上能力元信息

如果这部分配错，常见后果包括：

- 明明模型支持工具调用，但框架层不敢开
- 明明模型不支持结构化输出，却硬走 structured output
- 对兼容接口模型的能力推断不准确

#### `family` 应该怎么填

在官方语义里，`family` 更偏“模型家族”而不是“平台名”。

例如更自然的值通常是：

- `gpt-4o`
- `gpt-4`
- `claude-3-7-sonnet`
- `llama-4-scout`
- `unknown`

你当前代码里：

```python
family: str = "Qwen"
```

以及：

```python
family = "GPT" if provider == "openai" else provider.capitalize()
```

这种写法技术上可以运行，因为官方允许 `family` 是字符串；  
但从文档语义上讲，它更像“自定义家族标签”，不一定等价于 AutoGen 内建的 `ModelFamily` 常量。

如果后续要更严谨，可以考虑：

- OpenAI 官方模型：优先使用官方已知 family
- 第三方兼容模型：不确定时统一写 `unknown`
- 只有你非常明确知道模型族系时，再写更具体的 family

#### 创建好客户端后，调用时还要传什么

创建客户端只是第一步，真正发请求通常是调用：

- `await client.create(...)`
- `await client.create_stream(...)`

`ChatCompletionClient.create()` 在官方文档里的参数是：

- `messages`
- `tools=[]`
- `tool_choice="auto"`
- `json_output=None`
- `extra_create_args={}`
- `cancellation_token=None`

也就是说，创建客户端和发送请求是两层配置：

##### 第一层：构造客户端时传

适合放“默认配置”和“连接配置”：

- `model`
- `api_key`
- `base_url`
- `timeout`
- `max_retries`
- `model_info`
- 默认的 `temperature` / `top_p` / `max_tokens`

##### 第二层：每次请求时传

适合放“本次调用特有配置”：

- `messages`
- `tools`
- `tool_choice`
- `json_output`
- `extra_create_args`
- `cancellation_token`

#### `create()` 里的几个关键参数怎么理解

##### `messages`

模型上下文消息列表，通常由这些类型组成：

- `SystemMessage`
- `UserMessage`
- `AssistantMessage`
- `FunctionExecutionResultMessage`

##### `tools`

可供模型调用的工具列表。

如果要让模型做 function calling，这里就要传工具定义。

##### `tool_choice`

控制工具调用策略：

- `"auto"`：模型自己决定要不要用工具
- `"required"`：强制模型必须用工具
- `"none"`：关闭工具调用

##### `json_output`

这个参数很重要。

它可以传三种意思：

- `None`：普通文本输出
- `True/False`：开启或关闭 JSON mode
- 一个 `Pydantic BaseModel` 类型：启用 structured output

官方文档也建议，做 structured output 时，优先用 `json_output=你的Pydantic模型`，而不是手写 `response_format`

##### `extra_create_args`

这个参数是“单次调用覆盖项”。

常见可以放：

- `temperature`
- `max_tokens`
- `top_p`
- `frequency_penalty`
- `presence_penalty`

所以可以把它理解成：

- 构造函数里的参数：客户端默认值
- `extra_create_args`：某一次请求的临时覆盖值

#### `create_stream()` 比 `create()` 多了什么

`create_stream()` 除了 `create()` 那套参数，还多了：

- `max_consecutive_empty_chunk_tolerance`
- `include_usage`

它适合：

- 流式输出
- SSE
- 前端逐字显示

官方文档里也提到，流式场景如果想带 token usage，可以设置 `include_usage=True`，或者通过 `stream_options` 传给底层 OpenAI API。

#### 结构化输出、工具调用、图片输入分别依赖什么

这个问题和 `ModelInfo` 强相关。

##### 结构化输出

至少要保证：

- `ModelInfo.structured_output = True`
- 调用时使用 `json_output=YourPydanticModel`

##### JSON 输出

至少要保证：

- `ModelInfo.json_output = True`
- 同时 prompt 里要明确要求模型输出 JSON

##### 工具调用

至少要保证：

- `ModelInfo.function_calling = True`
- 调用时提供 `tools=[...]`

##### 图片输入

至少要保证：

- `ModelInfo.vision = True`
- `messages` 里传入图片内容

- `stream_options`

如果后续想增强你自己的封装层，可以优先把这些参数加进去。

#### 一个更贴近当前项目的推荐写法

如果你的目标是继续兼容 `siliconflow` / `dashscope compatible mode` / `openai` 这类 provider，比较稳妥的写法是：

```python
from autogen_core.models import ModelInfo
from autogen_ext.models.openai import OpenAIChatCompletionClient

def build_client(model: str, api_key: str, base_url: str) -> OpenAIChatCompletionClient:
    return OpenAIChatCompletionClient(
        model=model,
        api_key=api_key,
        base_url=base_url,
        timeout=30,
        max_retries=3,
        model_info=ModelInfo(
            vision=False,
            function_calling=True,
            json_output=True,
            family="unknown",
            structured_output=True,
        ),
    )
```

如果以后要支持视觉模型，再把：

- `vision=True`

如果以后要接不支持工具调用的模型，再把：

- `function_calling=False`

#### 如果后续要支持更多模型，配置上怎么想

建议把配置分成三层理解：

##### 第一层：连接信息

- `provider`
- `base_url`
- `api_key`

##### 第二层：模型选择

- `model`
- `family`

##### 第三层：能力声明

- `vision`
- `function_calling`
- `json_output`
- `structured_output`
- `multiple_system_messages`

这样你在 YAML 里设计配置时，就不会把“账号配置”和“模型能力”混在一起。

#### 一句话总结

对于你当前项目来说，可以把 AutoGen 这部分记成：

- `autogen_ext.models` 提供具体模型客户端
- `OpenAIChatCompletionClient` 不只接 OpenAI，也常被用来接 OpenAI-compatible 平台
- `ModelInfo` 用来声明模型能力，尤其对第三方兼容模型很关键
- 创建客户端时最核心的是 `model/api_key/base_url/model_info`
- 真正请求时再通过 `create()` 传 `messages/tools/json_output/extra_create_args`

如果以后你要扩展 `src/core/model_client.py`，优先补齐 `timeout`、`max_retries`、`temperature`、`top_p`、`max_tokens`、`reasoning_effort` 这些参数，收益会最大。

### AutoGen：按 AgentChat、Core、Extensions 三大板块系统理解

如果说前面已经把 AutoGen 的一些局部点位拆开看过，那么这里要补的是一张更完整的图。

你现在学习新版 AutoGen，最好不要再把它当成“一个包里什么都有”的旧式框架，而是明确拆成三层：

- `autogen_agentchat`：高层 API，负责“怎么快速搭一个能工作的单 Agent / 多 Agent 应用”
- `autogen_core`：底层框架，负责“消息、运行时、工具协议、上下文、记忆、事件驱动”
- `autogen_ext`：扩展实现，负责“模型客户端、向量记忆、MCP 工具、代码执行器、远程 runtime”等

对于当前仓库，这三层的关系可以先记成一句话：

```text
Extensions 提供外部能力实现
    -> AgentChat 把这些实现组装成 Agent / Team
    -> LangGraph 再把多个 Agent 步骤编排成业务流程
```

#### 先建立新版 AutoGen 的三层分工

可以先看一张对照表：

| 关注点 | 主要包 | 你最先接触到的对象 | 当前项目里的体现 |
| --- | --- | --- | --- |
| 单 Agent、群聊、终止条件、消息流 | `autogen_agentchat` | `AssistantAgent`、`BaseChatAgent`、`SelectorGroupChat`、`TextMentionTermination` | `src/agents/search_agent.py`、`src/agents/analyse_agent.py`、`src/agents/sub_writing_agent/writing_chatGroup.py` |
| 工具协议、记忆协议、模型上下文、取消令牌、运行时 | `autogen_core` | `FunctionTool`、`CancellationToken`、`Memory`、`ChatCompletionContext`、`RoutedAgent` | `src/agents/sub_writing_agent/retrieval_agent.py`、`src/agents/analyse_agent.py` |
| 模型客户端、向量记忆、MCP 工具、执行器 | `autogen_ext` | `OpenAIChatCompletionClient`、`ChromaDBVectorMemory`、`mcp_server_tools` | `src/core/model_client.py` |

所以你看当前仓库时，不要只记“项目用了 AutoGen”，而要进一步拆开：

- `search_agent`、`retrieval_agent`、`writing_chatGroup` 这些，主要是在用 `AgentChat`
- `FunctionTool`、`CancellationToken`、`Memory`、`model_context` 这些，是 `Core` 的概念
- `OpenAIChatCompletionClient` 这种具体连模型的类，是 `Extensions`

---

### AgentChat

`AgentChat` 是新版 AutoGen 里最适合先入门的一层。

它的定位非常明确：

- 它是高层 API
- 建立在 `autogen-core` 之上
- 适合快速构建单 Agent 和多 Agent 应用
- 对初学者和大多数业务项目来说，优先从它开始最合适

#### AgentChat 主要解决什么问题

如果你只关心：

- 我怎么创建一个 Agent
- 我怎么让 Agent 调模型
- 我怎么给 Agent 加工具
- 我怎么让多个 Agent 组成 Team
- 我怎么控制停止条件
- 我怎么拿到流式消息和最终结果

那么大部分场景都先用 `AgentChat` 就够了。

也就是说，`AgentChat` 帮你把很多底层细节先包好了：

- 消息怎么存
- 工具调用怎么循环
- Team 怎么驱动
- `run()` / `run_stream()` 怎么统一接口

所以当前项目很多地方直接用：

```python
from autogen_agentchat.agents import AssistantAgent
from autogen_agentchat.teams import SelectorGroupChat
from autogen_agentchat.conditions import TextMentionTermination
```

这是很自然的用法。

#### 从最小 AssistantAgent 开始

最常见的起点就是 `AssistantAgent`。

一个最小示意可以写成：

```python
from autogen_agentchat.agents import AssistantAgent
from autogen_ext.models.openai import OpenAIChatCompletionClient

model_client = OpenAIChatCompletionClient(model="gpt-4.1-nano")

agent = AssistantAgent(
    name="assistant",
    model_client=model_client,
    system_message="你是一个 helpful assistant"
)

result = await agent.run(task="解释一下 AutoGen 的 AgentChat")
print(result.messages[-1].content)
```

这个时候可以先抓住 4 个核心输入：

- `name`：Agent 名称
- `model_client`：底层模型客户端
- `system_message`：系统提示词
- `task`：当前一次任务输入

而输出通常是一个 `TaskResult`，其中 `messages[-1]` 一般就是最后的回复消息。

#### AgentChat 里的 Agent 是“有状态”的

这一点很关键。

官方文档对 `AgentChat` 的 Agent 有一个非常重要的说明：

- Agent 是有状态的
- `run()` / `run_stream()` 预期接收的是“新的消息”
- 不是每次都把完整历史重新传进去

这意味着：

```text
Agent 实例本身
    持有对话状态
    持有模型上下文
    持有工具调用相关上下文
```

所以你使用 `AssistantAgent` 时，应该这样理解：

- 同一个 agent 实例连续 `run(...)`，它会延续会话
- 如果你 `reset()`，会话状态就会清空
- 如果你创建一个新的 agent 实例，那就是新会话

这也是为什么当前项目里很多节点是“每次请求新建 Agent / Team”。

因为这等价于：

- 每个请求天然隔离
- 每个请求都是新的会话上下文
- 不容易把上一轮任务的上下文串进下一轮

#### 当前项目里最典型的 AgentChat 构造方式

当前仓库里可以看到几种很典型的 AgentChat 用法。

##### 普通 `AssistantAgent`

在 `src/agents/search_agent.py` 中：

```python
search_agent = AssistantAgent(
    name="search_agent",
    model_client=model_client,
    system_message=search_agent_prompt,
    output_content_type=SearchQuery
)
```

这里已经包含了两个关键点：

- 用 `AssistantAgent` 驱动模型生成
- 用 `output_content_type=SearchQuery` 约束结构化输出

##### 带工具的 `AssistantAgent`

在 `src/agents/sub_writing_agent/retrieval_agent.py` 中：

```python
retriever = FunctionTool(
    retrieval_tool,
    description="用于从本地知识库中查询外部资料，来辅助写作的工具"
)

retrieval_agent = AssistantAgent(
    name="retrieval_agent",
    model_client=model_client,
    system_message=retrieval_agent_prompt,
    tools=[retriever],
    reflect_on_tool_use=False
)
```

这里体现的是：

- AgentChat 可以直接挂工具
- 工具调用由 AgentChat 负责调度
- 返回结果可以直接作为工具摘要，也可以让模型再反思整理

##### 自定义 `BaseChatAgent`

在 `src/agents/analyse_agent.py` 中：

```python
class AnalyseAgent(BaseChatAgent):
    ...
```

这说明当前项目已经不只是“调用内置 Agent”，而是开始进入：

- 自己定义消息处理逻辑
- 自己决定内部如何串多个子 Agent
- 自己决定返回什么消息

这就是从“会用 AgentChat”走向“会扩展 AgentChat”的关键一步。

#### 结构化输出：AgentChat 里非常值得优先掌握的能力

新版 AutoGen 的一个很实用能力，是让 Agent 直接返回结构化结果。

当前项目的 `search_agent` 就是典型例子：

```python
class SearchQuery(BaseModel):
    querys: List[str]
    start_date: Optional[str]
    end_date: Optional[str]

search_agent = AssistantAgent(
    name="search_agent",
    model_client=model_client,
    system_message=search_agent_prompt,
    output_content_type=SearchQuery,
)
```

它的价值非常大：

- 不是只得到一段自然语言
- 而是直接得到可以被程序消费的数据结构
- 后续节点更容易校验、解析、调试

你可以把它理解成：

```text
自然语言任务
  -> Agent 推理
  -> 结构化对象
  -> 直接进入业务逻辑
```

这类模式非常适合：

- 检索条件生成
- 路由决策
- 审批结果
- 报告大纲
- 评审意见

#### 工具：AgentChat 里“让 Agent 真能做事”的关键

`AssistantAgent` 不只是会聊天，它还可以调工具。

当前项目已经在 `retrieval_agent` 里用了：

```python
from autogen_core.tools import FunctionTool
```

虽然 `FunctionTool` 来自 `Core`，但最常见的使用场景恰恰是在 `AgentChat` 里把它挂给 `AssistantAgent`。

最小模式可以记成：

```python
from autogen_agentchat.agents import AssistantAgent
from autogen_ext.models.openai import OpenAIChatCompletionClient

async def search_docs(query: str) -> str:
    """搜索知识库并返回结果"""
    return "..."

agent = AssistantAgent(
    name="doc_agent",
    model_client=OpenAIChatCompletionClient(model="gpt-4.1-nano"),
    tools=[search_docs],
    system_message="需要信息时优先使用工具"
)
```

这里有几个要点：

- 直接传 Python 函数也可以，AgentChat 会自动转换成 `FunctionTool`
- 也可以自己先手工构造 `FunctionTool`
- 工具 schema 会从函数签名和 docstring 推导出来
- 是否需要“工具调用后再总结一轮”，由 `reflect_on_tool_use` 控制

##### `reflect_on_tool_use` 怎么理解

默认情况下，Agent 执行工具后，往往会返回 `ToolCallSummaryMessage`。

如果工具输出本身就是可直接给用户看的自然语言，那么：

```python
reflect_on_tool_use=False
```

通常就够了。

如果工具输出更偏原始数据、JSON、检索片段、列表结果，那么：

```python
reflect_on_tool_use=True
```

会更合适，因为模型会在工具调用后再做一次整理和归纳。

这也是为什么你现在的 `retrieval_agent` 关闭了反思：

- 说明你更希望拿到直接工具结果
- 而不是再让模型多总结一轮

#### 多轮工具调用和并行工具调用

新版 AgentChat 里，工具不是只能调一次。

你需要知道两个点：

##### `max_tool_iterations`

它控制工具调用迭代次数。

例如：

```python
agent = AssistantAgent(
    name="assistant_loop",
    model_client=model_client,
    tools=[search_docs],
    max_tool_iterations=10,
)
```

表示：

- 模型可以反复决定要不要继续调用工具
- 直到不再需要工具，或者达到最大迭代次数

##### `parallel_tool_calls`

对 `OpenAIChatCompletionClient` 这类支持并行工具调用的模型客户端，可以设置：

```python
model_client = OpenAIChatCompletionClient(
    model="gpt-4o",
    parallel_tool_calls=False,
)
```

如果你希望调用顺序更可控，或工具之间有明显依赖关系，关闭并行会更稳。

#### Team：为什么多 Agent 协作不等于多个 Agent 手动串起来

很多人一开始会写成：

```text
agent_a.run(...)
agent_b.run(...)
agent_c.run(...)
```

这当然能跑，但它不等于 AgentChat 里的 Team。

Team 的核心是：

- 多个 Agent 共享一套协作上下文
- 有明确的团队运行器
- 有统一的终止条件
- 可以流式观察中间消息

当前项目最典型的 Team 是 `SelectorGroupChat`。

在 `src/agents/sub_writing_agent/writing_chatGroup.py` 中：

```python
task_group = SelectorGroupChat(
    [writing_agent, retrieval_agent, review_agent],
    model_client=model_client,
    termination_condition=text_termination,
    selector_prompt=selector_prompt,
    allow_repeated_speaker=False,
)
```

#### `SelectorGroupChat` 的工作方式

`SelectorGroupChat` 是新版 AgentChat 里非常重要的一个 Team 预设。

它的思路是：

- 多个 Agent 共享上下文
- 每轮只让一个 Agent 发言
- 下一位说话者由模型根据上下文动态选择

你可以把它理解成：

```text
共享消息线程
  -> selector model 选下一位发言者
  -> 被选中的 agent 执行一轮
  -> 写回共享上下文
  -> 再继续选择下一位
```

它特别适合：

- 规划者 + 执行者 + 审核者
- 写作者 + 检索者 + 评审者
- 分工明确，但谁先谁后不完全固定的场景

这正好对应你当前项目的写作小组：

- `writing_agent` 负责写
- `retrieval_agent` 负责补资料
- `review_agent` 负责审查

#### `SelectorGroupChat` 里最重要的 4 个配置

##### `participants`

也就是参与的 Agent 列表。

##### `model_client`

这个模型不是每个 Agent 自己发言的模型，而是“选择下一位发言者”时所用的模型。

##### `termination_condition`

例如你当前项目用的是：

```python
TextMentionTermination("APPROVE")
```

它表示：

- 当某条消息中出现 `APPROVE`
- 团队就终止

##### `selector_prompt`

它直接影响“下一位发言者怎么选”。

所以这个 prompt 非常关键。

如果 selector prompt 写不好，经常会出现：

- 一直选错 Agent
- 在不该结束时提前结束
- 某个 Agent 被频繁误选

#### 会话、状态与持久化：AgentChat 里要分成三层看

你提到“记忆会话”，在新版 AutoGen 里，至少要分成三层：

##### 第一层：短期会话状态

这是 Agent 或 Team 当前运行过程中保存的上下文。

比如：

- 当前这轮对话消息
- 已执行过的工具结果
- Team 的消息线程

这类状态默认保存在 Agent / Team 实例里。

##### 第二层：可保存/可恢复的会话状态

AgentChat 官方提供了：

- `save_state()`
- `load_state()`
- `reset()`

这意味着你可以：

- 跑完一轮后把状态导出来
- 存盘或存数据库
- 下一次请求再加载回来继续聊

例如：

```python
agent_state = await assistant_agent.save_state()
await new_assistant_agent.load_state(agent_state)
```

Team 也支持同样思路：

```python
team_state = await team.save_state()
await team.load_state(team_state)
```

这很适合：

- Web 应用无状态接口
- 多轮会话恢复
- 中断后继续任务

##### 第三层：外部记忆

这不是“把历史消息存起来”，而是：

- 另外维护一份 memory store
- 在每次模型调用前，检索相关记忆
- 注入进当前 `model_context`

这一层才更接近“长期记忆”或“RAG 记忆”。

#### `Memory` 和“普通聊天历史”不是一回事

这是新版 AutoGen 很容易混淆的点。

##### 普通聊天历史

它是：

- 用户说了什么
- Agent 回复了什么
- 工具调了什么

这些一般都在 Agent 的 `model_context` 里。

##### `Memory`

它是：

- 一份单独的记忆存储
- 可以按需查回
- 可以在每次推理前动态补进上下文

所以 `Memory` 更像：

```text
长期知识 / 用户偏好 / 历史事实 / RAG 检索层
```

而不是简单的 message history。

#### AgentChat 里怎么挂 Memory

官方提供的最简单例子是 `ListMemory`。

一个最小示意是：

```python
from autogen_agentchat.agents import AssistantAgent
from autogen_core.memory import ListMemory, MemoryContent, MemoryMimeType

user_memory = ListMemory()
await user_memory.add(
    MemoryContent(
        content="用户要求温度使用摄氏度",
        mime_type=MemoryMimeType.TEXT
    )
)

agent = AssistantAgent(
    name="assistant",
    model_client=model_client,
    tools=[get_weather],
    memory=[user_memory],
)
```

这个过程本质上是：

```text
收到任务
  -> 查询 memory
  -> 把相关记忆写入 model_context
  -> 再调用模型
```

所以 `Memory` 不是自己替代模型推理，而是给推理前补上下文。

#### `model_context`：短期上下文窗口控制

`AssistantAgent` 还有一个很重要的参数：

```python
model_context=...
```

它控制“本轮给模型看哪些消息”。

例如：

```python
from autogen_core.model_context import BufferedChatCompletionContext

agent = AssistantAgent(
    name="assistant",
    model_client=model_client,
    model_context=BufferedChatCompletionContext(buffer_size=5),
)
```

表示只保留最近 5 条消息作为模型上下文。

这对长会话很重要，因为它关系到：

- token 成本
- 历史污染
- 模型能不能抓住重点

所以你现在如果以后要做“连续对话研究助理”，会话通常要同时考虑两件事：

- `model_context` 负责短期上下文窗口
- `memory` 负责长期记忆召回

#### 自定义 Agent：什么时候要继承 `BaseChatAgent`

如果内置 `AssistantAgent` 不够用了，就进入自定义 Agent。

官方对 `BaseChatAgent` 的要求非常明确：

- 实现 `on_messages()`
- 实现 `on_reset()`
- 声明 `produced_message_types`
- 如果要流式输出，再实现 `on_messages_stream()`

一个最小骨架可以记成：

```python
from typing import Sequence
from autogen_agentchat.agents import BaseChatAgent
from autogen_agentchat.base import Response
from autogen_agentchat.messages import BaseChatMessage, TextMessage
from autogen_core import CancellationToken

class MyAgent(BaseChatAgent):
    def __init__(self, name: str = "my_agent"):
        super().__init__(name=name, description="自定义 agent")
        self._history = []

    @property
    def produced_message_types(self):
        return (TextMessage,)

    async def on_messages(self, messages: Sequence[BaseChatMessage], cancellation_token: CancellationToken) -> Response:
        self._history.extend(messages)
        reply = TextMessage(content="done", source=self.name)
        self._history.append(reply)
        return Response(chat_message=reply)

    async def on_reset(self, cancellation_token: CancellationToken) -> None:
        self._history.clear()
```

这里最重要的认知是：

- 自定义 Agent 自己维护内部状态
- `messages` 可能只是“新消息”
- 真正的连续性往往要靠实例字段自己保存

#### 当前项目里 `AnalyseAgent` 的意义

`AnalyseAgent(BaseChatAgent)` 在当前仓库里很有代表性。

它不是简单地“让一个模型回答问题”，而是：

```text
AnalyseAgent
  -> cluster_agent
  -> deep_analyse_agent
  -> global_analyse_agent
  -> 汇总成一个 Response
```

这说明它更像一个“Agent 内部编排器”。

也就是说：

- 对外，它是一个 AgentChat Agent
- 对内，它自己又调了多个子 Agent

这是一种非常重要的扩展模式。

### Core

如果说 `AgentChat` 负责“快”，那么 `Core` 负责“底层可控”。

`Core` 的定位不是让你更快写 demo，而是让你能构建：

- 事件驱动系统
- 分布式 agent system
- 强控制、多 runtime、多进程、多节点协作
- 更低层的消息协议和运行机制

#### Core 要解决什么问题

`Core` 更关心这些事情：

- Agent 如何收发异步消息
- 运行时如何管理 Agent 生命周期
- 消息如何按类型路由
- 工具协议如何抽象
- 模型客户端如何抽象
- 记忆、上下文、运行时如何解耦

所以 `Core` 的世界观更像：

```text
消息驱动
  + Actor 模型
  + Runtime 管理 Agent
  + Agent 通过 handler 处理不同类型消息
```

这和 `AgentChat` 的区别在于：

- `AgentChat` 是“我想要一个会工作的 Agent/Team”
- `Core` 是“我想自己定义底层运行方式”

#### Core 里最关键的几个概念

学习 `Core` 时，优先记住这些对象：

- `RoutedAgent`
- `message_handler`
- `AgentId`
- `SingleThreadedAgentRuntime`
- `CancellationToken`
- `ChatCompletionClient`
- `Tool` / `FunctionTool`
- `ChatCompletionContext`
- `Memory`

可以把它们对应成：

| Core 概念 | 作用 |
| --- | --- |
| `RoutedAgent` | 一个可按消息类型路由处理逻辑的 Agent |
| `message_handler` | 为某类消息注册处理函数 |
| `SingleThreadedAgentRuntime` | 本地单线程运行时，适合开发调试 |
| `CancellationToken` | 取消控制 |
| `ChatCompletionClient` | 模型客户端协议抽象 |
| `FunctionTool` | 把函数包装成工具协议对象 |
| `ChatCompletionContext` | 模型上下文存取抽象 |
| `Memory` | 记忆协议抽象 |

#### `RoutedAgent` 是 Core 里很核心的编程方式

在 Core 里，通常不是手工写一个 `if/else` 分发所有消息，而是：

- 继承 `RoutedAgent`
- 用 `@message_handler` 按消息类型注册处理器

一个简化理解如下：

```python
from dataclasses import dataclass
from autogen_core import RoutedAgent, message_handler, MessageContext

@dataclass
class Query:
    text: str

class SearchWorker(RoutedAgent):
    @message_handler
    async def handle_query(self, message: Query, ctx: MessageContext) -> None:
        ...
```

这样做的好处是：

- 不同消息类型各走各的 handler
- 更适合复杂系统
- 更容易接 runtime 和分布式场景

#### Runtime：Core 里 Agent 不是你手工直接“养着”的

这一点和 AgentChat 很不一样。

在 `Core` 里，Agent 通常由 runtime 管理。

比如本地开发常见的是：

```python
SingleThreadedAgentRuntime
```

它负责：

- 创建和注册 Agent
- 在 Agent 之间转发消息
- 管理生命周期
- 执行事件循环

所以 Core 的标准心智模型不是：

```python
agent.run(...)
```

而更像：

```text
runtime 管 agent
  -> agent 收消息
  -> handler 处理
  -> agent 再发消息
```

#### Core 里的工具：本质上是协议，不只是给 AgentChat 用

当前项目里你已经在用：

```python
from autogen_core.tools import FunctionTool
```

很多人第一次看到 `FunctionTool` 会以为它只是 `AssistantAgent(tools=[...])` 的附属品。

其实不是。

更准确地说：

- `FunctionTool` 是 Core 提供的工具协议实现
- AgentChat 只是把它拿来做高层封装

这意味着：

- 你可以在 `AssistantAgent` 里用它
- 也可以在 `Core` 自己写的 tool-equipped agent 里用它

#### Core 里的模型客户端抽象

`Core` 并不直接等于某个特定厂商模型。

它定义的是抽象接口，比如：

- `ChatCompletionClient`

然后具体实现交给 `Extensions` 提供。

所以这条链路要记清楚：

```text
autogen_core.models.ChatCompletionClient
    <- 抽象协议

autogen_ext.models.openai.OpenAIChatCompletionClient
    <- 具体实现
```

这也是为什么当前项目 `src/core/model_client.py` 明明创建的是：

```python
OpenAIChatCompletionClient
```

但它前面还会配一个：

```python
ModelInfo(...)
```

本质上是在把某个 OpenAI-compatible 接口适配进 Core 的统一模型能力抽象里。

#### Core 里的记忆与上下文

`Memory` 和 `ChatCompletionContext` 都属于 `Core` 层概念。

这里一定要区分：

##### `ChatCompletionContext`

它主要回答：

- 当前给模型看的消息集合是什么
- 消息如何保存
- 如何裁剪

##### `Memory`

它主要回答：

- 外部知识或偏好存哪里
- 需要时怎么检索
- 怎么把检索到的内容补进 `model_context`

也就是说：

```text
model_context = 当前上下文窗口
memory = 可检索补充的外部记忆层
```

这是新版 AutoGen 很重要的一条底层分界。

#### 什么时候应该从 AgentChat 往下沉到 Core

通常是下面这些时候：

### Extensions

如果说 `Core` 是抽象层，那么 `Extensions` 就是“把抽象接到真实世界”的实现层。

官方对 `autogen-ext` 的定位很明确：

- 它提供内建组件实现
- 包括模型客户端、工具、执行器、远程 runtime、memory store 等

#### Extensions 里最值得先掌握的 5 类能力

你可以先把 `Extensions` 记成 5 大类：

- `autogen_ext.models.*`：模型客户端
- `autogen_ext.tools.*`：工具扩展，包括 MCP
- `autogen_ext.memory.*`：向量记忆等记忆存储
- `autogen_ext.executors.*`：代码执行器
- `autogen_ext.runtimes.*`：远程/分布式 runtime

当前项目最直接依赖的是第一类：

- `autogen_ext.models.openai.OpenAIChatCompletionClient`

#### 模型客户端：当前项目最核心的 Extensions 用法

在 `src/core/model_client.py` 中：

```python
from autogen_ext.models.openai import OpenAIChatCompletionClient
from autogen_core.models import ModelInfo
```

然后封装成：

```python
return OpenAIChatCompletionClient(
    model=model,
    api_key=api_key,
    base_url=base_url,
    model_info=model_info
)
```

这说明你当前项目对 `Extensions` 的主要使用方式是：

```text
给 AutoGen 提供一个 OpenAI-compatible 的模型客户端
```

这里有两个关键层次：

##### 第一层：连接参数

- `model`
- `api_key`
- `base_url`

这是“连上哪个服务”。

##### 第二层：能力声明

```python
ModelInfo(
    vision=vision,
    function_calling=function_calling,
    json_output=json_output,
    family=family,
    structured_output=structured_output,
)
```

这是“告诉 AutoGen 这个模型具备哪些能力”。

所以 `model_client.py` 并不只是配置文件读取器，它本质上是：

- 对模型服务接入做适配
- 把业务里的模型能力转成 AutoGen 能理解的能力声明

#### 为什么 `OpenAIChatCompletionClient` 很关键

因为在新版 AutoGen 里，几乎所有高层能力最后都要落到它这种模型客户端上：

- Agent 的发言
- Selector 的下一位 speaker 选择
- 工具调用 schema 提交
- 结构化输出
- JSON 输出
- 流式生成

你可以把它理解成：

```text
AgentChat / Core 的很多抽象
    最终都要通过 model client 才能真的调用模型
```

#### ChromaDB 是什么

ChromaDB 可以先粗略理解成：

- 一个专门存“文本 + 向量 + 元数据”的数据库
- 它最擅长做语义相似检索
- 常用于 RAG、知识库问答、长文档分块召回、多 Agent 共享知识源

和普通关系型数据库相比：

- MySQL / PostgreSQL 更擅长结构化条件查询
- ChromaDB 更擅长“这个问题和哪些文本语义最接近”

所以它最典型的工作流是：

```text
原始文本
-> 切分为多个 chunk
-> 生成 embedding
-> 存入 Chroma collection
-> 用户提问
-> 对 query 做 embedding
-> 找最相近的 top-k chunk
-> 把 chunk 返回给应用或 LLM
```

一句话记忆：

> ChromaDB 是一个以 `collection` 为基本单位、支持 embedding 存储与语义检索的向量数据库。

官方入口：

- Chroma 官方文档  
  <https://docs.trychroma.com/>
- Chroma Python Reference  
  <https://docs.trychroma.com/reference/python/client>

#### ChromaDB 的核心对象

理解 Chroma 时，最重要的是先分清 4 个东西：

1. `Client`
2. `Collection`
3. `Record`
4. `Embedding Function`

##### `Client`

`Client` 是你和 ChromaDB 交互的入口。

它负责：

- 创建 collection
- 获取 collection
- 删除 collection
- 枚举 collection
- 在某些模式下管理 database / tenant

你可以把它理解成：

```text
数据库连接对象 + collection 管理入口
```

##### `Collection`

`Collection` 是 Chroma 里的基本存储单位。

可以把它近似理解成：

- 一个“向量表”
- 或一个“语义检索库”

一条记录通常包含：

- `id`
- `document`
- `embedding`
- `metadata`

例如一条记录可以想成：

```python
{
    "id": "doc1_chunk_0",
    "document": "RAG combines retrieval and generation.",
    "embedding": [0.12, 0.44, ...],
    "metadata": {
        "source": "rag.md",
        "chunk": 0
    }
}
```

##### `documents / metadatas / ids / embeddings`

这是你最常在 `collection.add(...)` 和 `collection.upsert(...)` 里看到的 4 组数据：

- `documents`
  - 原始文本内容
- `metadatas`
  - 附加信息，如来源、页码、标签、chunk_id
- `ids`
  - 每条记录的唯一标识
- `embeddings`
  - 文本对应的向量

一个非常重要的点：

- 这四类数据本质上是按位置一一对应的
- 列表长度必须匹配

例如：

```python
documents = ["文本A", "文本B"]
metadatas = [{"source": "a.md"}, {"source": "b.md"}]
ids = ["id_a", "id_b"]
```

它的含义就是：

- 第 0 条文本 `"文本A"` 对应 metadata `{"source": "a.md"}` 和 id `id_a`
- 第 1 条文本 `"文本B"` 对应 metadata `{"source": "b.md"}` 和 id `id_b`

##### 一个最小 record 示例

```python
import chromadb

client = chromadb.Client()
collection = client.get_or_create_collection(name="demo")

collection.add(
    documents=[
        "RAG combines retrieval and generation.",
        "Embeddings convert text into vectors."
    ],
    metadatas=[
        {"source": "rag_note.md", "chunk": 0},
        {"source": "rag_note.md", "chunk": 1}
    ],
    ids=[
        "rag_note_chunk_0",
        "rag_note_chunk_1"
    ]
)
```

这段代码的意思是：

- 往 `demo` 这个 collection 里插入两条记录
- 每条记录都带文本、元数据和唯一 id
- 如果 collection 已配置 embedding function，Chroma 会自动帮你算向量

#### 客户端创建与持久化方式

这一部分重点回答：

- Chroma 有哪些 client
- 哪些数据会落磁盘
- 哪些只存在内存

官方入口：

- Python Client Reference  
  <https://docs.trychroma.com/reference/python/client>

##### `chromadb.Client()`

这是最基础的 client，用于本地直接创建并操作 Chroma。

最小示例：

```python
import chromadb

client = chromadb.Client()
collection = client.create_collection(name="my_collection")
```

你可以把它理解成：

- 一个最直接的本地 client
- 适合 demo、测试、理解 API

##### `EphemeralClient`

`EphemeralClient` 是显式“只保存在内存里”的 client。

特点：

- 数据不持久化
- 进程结束后数据就消失
- 适合临时实验、单元测试、notebook demo

示例：

```python
import chromadb

client = chromadb.EphemeralClient()
collection = client.get_or_create_collection(name="temp_demo")
```

`PersistentClient` 是当前工程项目里最常用的一种方式。

特点：

- 数据落到本地目录
- 重启后还能继续使用
- 最适合本地知识库、开发环境、小规模部署

示例：

```python
import chromadb

client = chromadb.PersistentClient(path="./chroma_data")
collection = client.get_or_create_collection(name="notes")
```

这也是当前项目主线使用的方式。

##### `HttpClient`

当 Chroma 以服务方式部署时，可以用 `HttpClient` 连接远端服务器。

这时你的应用和 Chroma 是分离的：

- 应用程序在本地或某台服务机器
- Chroma server 单独跑在某个 host/port

```python
import chromadb

client = chromadb.HttpClient(host="localhost", port=8000)
```

#### Embedding 如何接入

这一部分重点回答 3 个问题：

1. Chroma 会不会自动算 embedding
2. 如果我自己算好了 embedding 怎么传
3. 同一个 collection 能不能混用不同 embedding 模型

##### collection 绑定 embedding function

如果 collection 在创建时绑定了 embedding function，那么你只传 `documents` 就可以：

```python
collection = client.get_or_create_collection(
    name="demo_with_ef",
    embedding_function=my_embedding_function
)

collection.add(
    ids=["id1"],
    documents=["Chroma will compute embeddings for this text."]
)
```

这时 Chroma 会：

- 用 collection 的 embedding function 计算 embedding
- 然后把 `documents + embeddings + metadatas + ids` 一起存进去

##### 显式传 `embeddings`

如果你已经自己算好了向量，也可以显式传：

```python
collection.add(
    ids=["id1"],
    documents=["Manually embedded document"],
    embeddings=[[0.12, 0.98, 0.33]]
)
```

这时 Chroma 不会再重新算 embedding。

##### 维度相同，能不能换不同 embedding 模型

这里要给一个非常明确的工程结论：

- **理论上**：如果向量维度相同，数据库层面未必立即报错
- **工程上**：**不建议混用**

原因是：

- Chroma 真正存的是向量，不是模型名
- 但不同 embedding 模型即使维度一样，向量空间的语义分布也可能完全不同
- 混用后，距离计算会失去可比性，检索质量会明显变差

所以更合理的实践是：

1. 一个 collection 固定使用一套 embedding 模型
2. 如果要换模型：
   - 新建 collection
   - 全量重建 embeddings
   - 再迁移使用

也就是说：

> 同维度不等于同语义空间。维度一致只是“形状兼容”，不代表“检索质量兼容”。

#### 文本分块如何实现

这一部分专门回答：

- 为什么要分块
- 当前项目支持哪些分块方式
- 参数怎么影响 chunk

##### 为什么要分块

长文本通常不直接整篇存进向量数据库，原因有三：

1. 一整篇文本太长，embedding 语义会被平均，检索粒度太粗
2. 用户问题往往只对应文档里的某一小段
3. RAG 最终喂给 LLM 的上下文通常也是 chunk 级别

所以分块的目标是：

- 让检索结果更精确
- 让召回内容更适合直接送给 LLM

一句话：

> chunk 是 RAG / semantic retrieval 的基本单位。

```python
    similarity_threshold=config.get_float("tmpdb_similarity_threshold")
)
```

查询结果再被 `retrieval_tool` 整理成可供 Agent 使用的文本返回。

##### 当前项目里的实际调用链

你可以把当前项目里 ChromaDB 的使用过程记成：

```text
reading_node
-> create_database("临时知识库")
-> add_processed_content(...)
-> ChromaDB collection

writing_node 内部的 retrieval_agent
-> retrieval_tool(...)
-> knowledge_base.aquery(...)
-> 从临时知识库 / 用户知识库取回相关 chunk
```

#### 本节结论

把这一节压缩成几句最值得记的话：

1. `Collection` 是 Chroma 里的基本存储单位，本质上像“向量表”
2. `documents / metadatas / ids / embeddings` 是 Chroma record 的核心组成
3. 工程里最常用的是 `PersistentClient + get_or_create_collection + add/query`
4. 一个 collection 应尽量固定使用同一套 embedding 模型，不建议混用同维度但不同模型的向量
5. 当前项目不是直接裸调 Chroma，而是通过：

```text
ChromaKB -> KnowledgeBase -> KnowledgeBaseManager -> knowledge_base
```

做了统一封装

6. 当前项目里 Chroma 的主职责是：

- 阅读阶段存结构化论文知识
- 写作阶段做检索增强

前四章分别讲单个框架与组件，这一章把它们重新串回 `Paper-Agent` 当前项目中，重点回答：整体技术栈怎么协作、从请求进入到报告输出的主调用链是什么、节点到 Agent 到知识库是如何连起来的，以及推荐按什么顺序去学习和阅读源码。

### 当前项目整体技术架构一句话说明

当前项目本质上是一个 **FastAPI 提供接口与流式输出、LangGraph 编排主流程、AutoGen 承担节点内智能行为、ChromaDB 提供知识库存储与检索增强** 的多模块协作系统。

### 各框架在当前项目中的职责分工

- `FastAPI`：暴露 HTTP API、处理上传与查询、通过 SSE 推送执行状态
- `LangGraph`：定义主流程图，控制节点顺序、条件流转与结束条件
- `AutoGen`：在节点内部构造 Agent、Team、工具调用和多 Agent 协作
- `ChromaDB`：存结构化知识和 chunk，供后续检索增强使用

### 从请求进入到报告输出的主链路

```text
前端请求 /api/research
-> FastAPI 创建 state_queue 与 orchestrator
-> LangGraph 进入 search_node
-> reading_node
-> analyse_node
-> writing_node
-> report_node
-> 最终输出 Markdown 报告，并通过 SSE 持续推送中间状态
```

### 节点到 Agent 到知识库的关系

- `search_node`：主要调用搜索相关 Agent 生成检索条件
- `reading_node`：并行阅读论文，抽取结构化结果，并把结果写入 Chroma 临时知识库
- `analyse_node`：由分析总控 Agent 串起聚类、深度分析和全局分析
- `writing_node`：每个章节内部起一个多 Agent 写作 Team，需要时通过 `retrieval_tool` 从 ChromaDB 检索资料
- `report_node`：汇总章节内容，生成最终报告

### 从“抽象”到“项目落地”

把 `AgentChat`、`Core`、`Extensions` 放在一起，你可以得到一条非常清晰的实现链路：


当前项目更像：

```text
LangGraph state
```

- `retrieval_tool`

### LlamaIndex 是什么

LlamaIndex 可以先粗略理解成：

- 一个围绕 **LLM + 数据** 构建的框架
- 它最核心的强项不是 Web API，也不是通用状态图，而是：
  - 文档读取
  - 文档转节点
  - 建索引
  - 检索
  - 查询引擎
  - 工具化
  - Agent 与 Workflow

你可以把它想成一个比较完整的“数据到回答”的框架：

```text
Reader
-> Document
-> Node
-> Index
-> Retriever
-> QueryEngine
-> Tool
-> Agent / Workflow
```

如果用一句话总结：

> LlamaIndex 是一个面向 LLM 应用的数据框架，重点解决“如何把外部数据可靠地组织、索引、检索，并让 Agent 或 Workflow 用起来”。

官方入口：

- Docs 首页：<https://docs.llamaindex.ai/en/stable/>

### 官方设计的核心功能总览

从官方文档结构看，LlamaIndex 的核心功能大致可以按下面 10 层理解：

1. **Loading**
   - 把外部文件、目录、数据库、API 数据读成 `Document`
2. **Documents / Nodes**
   - 用 `Document` 表示原始文档，用 `Node` 表示更适合检索的最小知识单元
3. **Transformations / Ingestion Pipeline**
   - 做切分、清洗、抽取标题、打 embedding、去重、缓存
4. **Indexes**
   - 把节点组织成可检索的数据结构
5. **Retrievers**
   - 决定如何从索引里召回相关节点
6. **QueryEngine**
   - 在召回之上做 response synthesis，形成最终回答
7. **ChatEngine**
   - 在 QueryEngine 基础上增加对话态
8. **Tools / Agents**
   - 把查询引擎、函数、服务封成工具，再让 Agent 调用
9. **Multi-agent**
   - 多个 Agent 通过 handoff 或 orchestrator 协作
10. **Workflow**
    - 用事件驱动方式编排更复杂的流程

这套设计和当前项目已有技术栈的关系可以先记成：

- `LangGraph`：更偏通用状态图编排
- `AutoGen`：更偏多 Agent 对话与工具协作
- `ChromaDB`：更偏底层向量存储
- `LlamaIndex`：更偏数据 -> 索引 -> 检索 -> QueryEngine -> Agent 的一体化框架

### LlamaIndex 的核心数据流

这一套框架最值得背下来的一条主线是：

```text
SimpleDirectoryReader / 自定义 Reader
-> Document
-> Node Parser / SentenceSplitter
-> Node
-> IngestionPipeline
-> VectorStoreIndex / SummaryIndex / KeywordTableIndex
-> Retriever
-> QueryEngine
-> Tool
-> Agent / Workflow
```

换句话说，LlamaIndex 并不是直接让你“拿文本问模型”，而是鼓励你先把数据组织成下面几层：

- 原始文档层
- 节点层
- 索引层
- 检索层
- 查询与响应层
- Agent / Workflow 调度层

这也是为什么它在 RAG 场景里特别自然。

`vector_store.py`

```python
import chromadb

from llama_index.vector_stores.chroma import ChromaVectorStore
from llama_index.core import StorageContext


def build_chroma_vector_store(persist_dir: str, collection_name: str):
    client = chromadb.PersistentClient(path=persist_dir)
    collection = client.get_or_create_collection(collection_name)
    vector_store = ChromaVectorStore(chroma_collection=collection)
    storage_context = StorageContext.from_defaults(vector_store=vector_store)
    return client, collection, vector_store, storage_context
```

`ingestion_pipeline.py`

```python
from llama_index.core.extractors import TitleExtractor
from llama_index.core.ingestion import IngestionPipeline, IngestionCache
from llama_index.core.node_parser import SentenceSplitter
from llama_index.embeddings.openai import OpenAIEmbedding


def build_indexing_pipeline(vector_store):
    cache = IngestionCache()
    return IngestionPipeline(
        transformations=[
            SentenceSplitter(chunk_size=800, chunk_overlap=120),
            TitleExtractor(nodes=5),
            OpenAIEmbedding(model="text-embedding-3-small"),
        ],
        vector_store=vector_store,
        cache=cache,
    )


def ingest_documents(documents, vector_store):
    pipeline = build_indexing_pipeline(vector_store)
    pipeline.run(documents=documents, num_workers=2)
```

- Chroma 持久化
- 可复用的 pipeline builder
- embedding 作为 transformation 阶段显式存在
- 可并行 ingest

### Indexes

官方入口：

- VectorStoreIndex：<https://docs.llamaindex.ai/en/stable/module_guides/indexing/vector_store_index/>

#### 官方设计里最值得先掌握的 Index

LlamaIndex 的 Index 不止一个，但在真实业务里最常用的是下面三类：

1. `VectorStoreIndex`
2. `SummaryIndex`
3. `KeywordTableIndex`

其中最重要的几乎总是：

- `VectorStoreIndex`

因为它直接对应大多数 RAG 场景。

#### 高频 API

必须优先掌握这些：

- `VectorStoreIndex.from_documents(documents, ...)`
- `VectorStoreIndex(nodes=..., storage_context=...)`
- `VectorStoreIndex.from_vector_store(vector_store, ...)`
- `index.as_retriever(...)`
- `index.as_query_engine(...)`

补充了解：

- `SummaryIndex(...)`
- `KeywordTableIndex(...)`

#### `VectorStoreIndex` 的两种常见创建方式

##### 方式 1：直接从文档创建

```python
index = VectorStoreIndex.from_documents(documents)
```

#### QueryEngine 是什么

`ChatEngine` 可以理解成：

- `QueryEngine` 的对话态版本

也就是：

- `query` 更像单轮问答

常见 API：

- `index.as_chat_engine(...)`

`workflow.py`

```python
from llama_index.core.workflow import Workflow, StartEvent, StopEvent, step, Context
from llama_index.llms.openai import OpenAI

from events import RetrieveEvent, RetrievedContextEvent, SynthesisEvent


class ResearchWorkflow(Workflow):
    def __init__(self, query_service, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self.query_service = query_service
        self.llm = OpenAI(model="gpt-4o-mini", temperature=0.1)

    @step
    async def start(self, ctx: Context, ev: StartEvent) -> RetrieveEvent:
        user_query = ev.topic
        return RetrieveEvent(query=user_query)

    @step
    async def retrieve(self, ctx: Context, ev: RetrieveEvent) -> RetrievedContextEvent:
        response = await self.query_service.aquery(ev.query)
        return RetrievedContextEvent(
            query=ev.query,
            context=str(response),
        )

    @step
    async def synthesize(self, ctx: Context, ev: RetrievedContextEvent) -> SynthesisEvent:
        prompt = (
            f"问题：{ev.query}\n"
            f"上下文：{ev.context}\n"
            "请基于上下文生成一段结构化研究结论。"
        )
        draft = self.llm.complete(prompt).text
        return SynthesisEvent(query=ev.query, draft=draft)

    @step
    async def finish(self, ctx: Context, ev: SynthesisEvent) -> StopEvent:
        return StopEvent(result={"query": ev.query, "answer": ev.draft})
```

### 与 LangGraph / AutoGen / ChromaDB 的边界

这一节只做定位说明，不展开选型大战。

#### LlamaIndex vs LangGraph

- `LangGraph` 更偏通用状态图编排
- `LlamaIndex Workflow` 更偏数据 / 检索 / Agent 生态内的事件驱动流程

如果你的重点是：

- 任意业务节点编排
- 状态图控制
- 复杂条件流转

那么 LangGraph 更自然。

如果你的重点是：

- 数据摄取
- 索引检索
- QueryEngine
- Agent 和 Workflow 直接围绕知识库工作

那么 LlamaIndex 更自然。

#### LlamaIndex vs AutoGen

- `AutoGen` 更偏多 Agent 消息协作、Team、Selector、对话驱动
- `LlamaIndex` 更偏“数据 -> 索引 -> 检索 -> QueryEngine -> Agent”一体化

所以它们的重心并不完全重合。

#### LlamaIndex vs ChromaDB

- `ChromaDB` 是底层向量数据库
- `LlamaIndex` 是其上的上层框架

你可以把关系理解成：

```text
ChromaDB 负责存和查向量
LlamaIndex 负责让这些向量真正进入“文档 -> 检索 -> 回答 -> Agent”的应用链路
```

## 第七章：Python 开发环境与依赖管理

本章由原 `docs/学习文档.md` 整理并入，统一收口到知识文档中维护。它不直接讨论 `PaperChatAgent` 的业务逻辑，而是补齐 Python 开发里最容易混淆的一层基础：包到底下载到哪里、实际安装到哪里、虚拟环境如何隔离、不同项目为什么不会互相污染，以及 `pip`、`venv`、`uv`、`poetry`、`conda` 各自应该在什么场景下使用。

建议先记住一个总原则：很多时候要同时分清楚两个位置。

- 下载缓存位置：安装包、wheel、构建缓存放哪里
- 实际安装位置：真正可以被 `import` 的包放哪里

这两个位置经常不是同一个地方。

### 7.1 `pip` 是什么、管理什么

`pip` 是 Python 官方生态里最基础、最通用的包安装工具。它主要负责把 PyPI 上的包安装到当前 Python 解释器对应的环境里。

#### 概念

`pip` 主要管理的是：

- Python 包
- 依赖版本解析
- 从 PyPI 下载包并安装到当前环境

`pip` 不直接负责这些事情：

- 自动创建虚拟环境
- 自动给每个项目做依赖隔离
- 标准化管理整个项目的依赖声明和锁文件
- 管理 Python 解释器本身

所以 `pip` 更像“安装器”，而不是完整的项目管理器。

#### 工作机制

执行下面这类命令时：

```bash
python -m pip install requests
python -m pip install "fastapi>=0.110"
python -m pip install -r requirements.txt
```

`pip` 会大致完成这些步骤：

1. 读取你指定的包和版本约束。
2. 解析依赖关系。
3. 下载 wheel 或源码包。
4. 构建并安装到当前 Python 环境的 `site-packages`。

#### mac 下的路径理解

使用 `pip` 时要分两种位置：

1. 下载缓存放哪里
2. 实际包装到哪里

在 macOS 下，`pip` 默认缓存常见位置是：

```bash
~/Library/Caches/pip
```

但真正能被 `import` 的包，不一定在这里，而是装进当前 Python 环境对应的 `site-packages`。

如果当前不在虚拟环境里，直接执行：

```bash
pip install xxx
```

那么包装到“当前 Python”的 `site-packages` 里，而不是装进某个项目目录。

#### 隔离方式

`pip` 自己不做项目隔离。
如果两个项目共用同一个 Python 环境，那么它们装的包就会互相影响。

所以 `pip` 最稳妥的常见用法是配合 `venv`：

```bash
python -m venv .venv
source .venv/bin/activate
python -m pip install fastapi
```

#### 常用命令

```bash
python -m pip install requests
python -m pip install -r requirements.txt
python -m pip uninstall requests
python -m pip list
python -m pip show requests
python -m pip freeze
python -m pip cache dir
```

#### 适用场景

`pip` 适合：

- 简单脚本或小项目
- 教程示例
- 需要最大兼容性的环境
- 已经明确知道当前 Python 环境来源时的直接安装

如果是认真维护的项目，通常建议和 `venv`、`uv` 或 `poetry` 搭配使用，而不是长期裸用系统环境。

### 7.2 `python -m venv .venv` 的作用与目录结构

`venv` 是 Python 标准库自带的虚拟环境方案，用来给项目创建独立的 Python 运行环境。

#### 概念

这条命令：

```bash
python -m venv .venv
```

含义是：在当前目录下创建一个叫 `.venv` 的虚拟环境。

这个目录里会包含：

- 一套独立的 Python 可执行文件
- 一套独立的 `pip`
- 一套独立的 `site-packages`

#### 工作机制

创建完成后，项目目录通常会多出：

```text
项目目录/
├── .venv/
│   ├── bin/
│   ├── lib/python3.x/site-packages/
│   └── pyvenv.cfg
```

激活环境后：

```bash
source .venv/bin/activate
```

后续执行的 `python` 和 `pip` 就优先指向 `.venv` 里的那一套。

#### mac 下的路径理解

当你在项目根目录执行：

```bash
python -m venv .venv
```

虚拟环境就创建在当前项目下的：

```bash
./.venv
```

如果之后安装依赖：

```bash
python -m pip install fastapi
```

那么包会进入：

```bash
.venv/lib/pythonX.Y/site-packages
```

这和“直接装到当前 Python 的全局 `site-packages`”是两回事。

#### 隔离方式

`venv` 的隔离本质是：

- 项目 A 有自己的 `.venv`
- 项目 B 有自己的 `.venv`
- 两边的 `site-packages` 分开

这样即使项目 A 用 `fastapi==0.110`，项目 B 用 `fastapi==0.115`，也可以互不影响。

#### 常用命令

```bash
python -m venv .venv
source .venv/bin/activate
deactivate
python -m pip install -U pip
python -m pip install -r requirements.txt
```

#### 适用场景

`venv` 适合：

- 希望使用最原生、最轻量的 Python 隔离方式
- 项目不需要额外的项目管理器
- 想和 `pip` 组合成传统但稳定的工作流

### 7.3 `uv` 如何管理依赖、缓存、Python 版本、项目隔离

`uv` 是新一代 Python 包与项目管理工具，它比 `pip` 更完整，既能管依赖，也能管虚拟环境、锁文件、Python 版本和命令运行。

#### 概念

`uv` 的定位不是“每个项目一套 uv”，而是：

- 机器上通常只有一个全局 `uv`
- `uv` 可以管理多个 Python 版本
- `uv` 可以管理多个项目
- 每个项目再拥有自己的独立 `.venv`

也就是说，`uv` 是全局工具；项目隔离靠的不是“多个 uv”，而是“多个项目环境”。

#### 工作机制

常见项目流转方式：

```bash
uv init myproj
cd myproj
uv add fastapi
uv add pytest --dev
uv run python main.py
uv sync
```

`uv` 会维护几个核心部分：

- `pyproject.toml`：项目依赖声明
- `uv.lock`：精确锁定版本
- `.venv`：当前项目自己的运行环境

`uv run` 会优先寻找当前项目对应的环境，而不是随便调用系统 Python。

#### mac 下的路径理解

在 `uv` 的语境里，要分清楚三层位置：

1. `uv` 工具本身
2. `uv` 全局管理的 Python
3. 项目自己的 `.venv`

其中：

- 全局 `uv` 是一份工具，可供所有项目共用
- 全局 Python 会被 `uv` 统一保存和复用
- 每个项目自己的依赖仍然装在各自的 `.venv` 里

此外，`uv` 还有全局缓存目录，负责缓存下载结果、构建结果等内容。

#### 隔离方式

`uv` 的项目隔离依赖于：

- 每个项目独立的 `.venv`
- 每个项目自己的 `pyproject.toml`
- 每个项目自己的 `uv.lock`

所以项目 A 和项目 B 即使都由同一个 `uv` 管理，依赖也不会混在一起。

可以把 `uv` 的职责拆成三层来记：

- 全局工具层：`uv`
- 全局资源层：`uv` 管理的 Python、缓存
- 项目隔离层：每个项目自己的 `.venv`

#### 常用命令

```bash
uv init
uv add fastapi
uv add pytest --dev
uv remove fastapi
uv sync
uv run python main.py
uv venv
uv cache dir
uv python dir
uv python list --only-installed
```

#### 适用场景

`uv` 适合：

- 新的 Python 项目
- 希望统一管理依赖、锁文件、虚拟环境、Python 版本
- 想保留接近 `pip` 的使用习惯，但需要更现代的工作流
- 希望项目目录结构清晰，并默认使用项目内 `.venv`

### 7.4 `poetry` 如何管理依赖与虚拟环境

`poetry` 是以“项目”为中心的 Python 依赖管理和打包工具。

#### 概念

`poetry` 主要管理：

- `pyproject.toml`
- `poetry.lock`
- 项目虚拟环境
- 构建与发布流程

相比 `pip`，`poetry` 更强调“先声明项目依赖，再统一解析、锁定、安装”。

#### 工作机制

常见流程如下：

```bash
poetry new myproj
cd myproj
poetry add fastapi
poetry add pytest --group dev
poetry install
poetry run python main.py
```

这里：

- 依赖声明写入 `pyproject.toml`
- 精确解析结果写入 `poetry.lock`
- 安装发生在 Poetry 管理的虚拟环境里

#### mac 下的路径理解

`poetry` 默认会把虚拟环境放在自己的缓存目录下，而不是直接放进项目目录。

常见默认位置是：

```bash
~/Library/Caches/pypoetry/virtualenvs
```

也可以改成项目内 `.venv`，这样每个项目的环境就和代码放在一起，更直观。

#### 隔离方式

`poetry` 的隔离本质上也是“每个项目一个虚拟环境”。

差别只在于默认环境位置：

- 默认：集中放在 Poetry 缓存目录下
- 可配置：放到项目根目录的 `.venv`

推荐在长期维护项目中开启项目内虚拟环境：

```bash
poetry config virtualenvs.in-project true
```

这样项目目录下就会出现：

```text
项目目录/
├── pyproject.toml
├── poetry.lock
└── .venv/
```

#### 常用命令

```bash
poetry add fastapi
poetry add pytest --group dev
poetry remove fastapi
poetry install
poetry update
poetry run python main.py
poetry env info --path
poetry env list --full-path
poetry config --list
```

#### 适用场景

`poetry` 适合：

- 团队协作项目
- 需要稳定依赖锁定
- 需要构建和发布 Python 包
- 希望围绕 `pyproject.toml` 做规范化管理

### 7.5 `conda` 的定位与和前面三者的区别

`conda` 和 `pip`、`uv`、`poetry` 的差别，核心不在命令写法，而在管理层级。

#### 概念

`conda` 不只是 Python 包管理器，它更像：

- 环境管理器
- Python 版本管理器
- 二进制依赖管理器

它不只会装 Python 包，还经常负责安装底层依赖，例如科学计算、机器学习、CUDA 相关组件。

#### 工作机制

常见用法：

```bash
conda create -n myenv python=3.11
conda activate myenv
conda install numpy pandas
```

`conda` 会创建一个独立环境，并把指定 Python 版本和相关依赖一起装进去。

#### mac 下的路径理解

`conda` 环境通常放在 Conda 自己管理的环境目录下。
如果使用的是 Anaconda，常见形式类似：

```bash
/opt/anaconda3/envs/环境名
```

这意味着 Conda 更像是“整套运行时环境”的隔离，而不是只关心 `site-packages`。

#### 隔离方式

`conda` 的隔离方式比 `venv` 更完整：

- 独立环境目录
- 独立 Python 版本
- 独立包集合
- 独立二进制依赖

所以它特别适合依赖复杂、底层库较重的场景。

#### 常用命令

```bash
conda create -n myenv python=3.11
conda activate myenv
conda install numpy pandas
conda list
conda env list
conda remove -n myenv --all
```

#### 适用场景

`conda` 适合：

- 数据科学
- 深度学习
- 需要 CUDA 或复杂二进制依赖
- 需要同时管理 Python 解释器和底层依赖

和前面三者相比，可以这样粗略区分：

- `pip`：最基础的包安装器
- `uv`：更现代的一体化项目管理工具
- `poetry`：偏项目依赖管理和打包发布
- `conda`：偏完整运行环境和底层依赖管理

### 7.6 4 个工具的选型建议

选择时，不要只看“哪个命令更好记”，而要先看自己要管理的对象到底是什么。

#### 通用选择思路

- 只是快速装包，且能明确当前 Python 来源：`pip`
- 想要传统稳定工作流：`pip + venv`
- 新项目，希望依赖、虚拟环境、锁文件、Python 版本统一管理：`uv`
- 团队项目，尤其需要打包和发布：`poetry`
- 科学计算、深度学习、CUDA、复杂二进制依赖：`conda`

#### 简单对比

| 工具 | 主要管理对象 | 默认是否强调项目隔离 | 是否适合管理 Python 版本 | 是否擅长二进制依赖 |
| --- | --- | --- | --- | --- |
| `pip` | 当前环境中的 Python 包 | 否 | 否 | 否 |
| `venv` | 项目级虚拟环境 | 是 | 否 | 否 |
| `uv` | 项目、依赖、环境、锁文件、Python 版本 | 是 | 是 | 否 |
| `poetry` | 项目依赖、锁文件、虚拟环境、打包 | 是 | 否 | 否 |
| `conda` | 环境、Python 版本、包、底层依赖 | 是 | 是 | 是 |

#### 推荐实践

如果是普通 Python 新项目，优先考虑 `uv`。
如果是传统兼容项目，可以稳定使用 `pip + venv`。
如果项目需要打包发布，`poetry` 依然是很常见的选择。
如果是科学计算、CUDA、底层依赖复杂的环境，优先使用 `conda`。
