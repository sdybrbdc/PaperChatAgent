## 第一章：Python 包与环境管理

这一章只整理 `pip`、`venv`、`uv`、`poetry`、`conda` 这几类工具。重点不是项目业务，而是先把 Python 开发里最容易混淆的几件事理顺：包下载到哪里、包实际安装到哪里、虚拟环境如何隔离、不同项目为什么不会互相污染。

建议先记住一个总原则：很多时候要同时分清楚两个位置。

- 下载缓存位置：安装包、wheel、构建缓存放哪里
- 实际安装位置：真正可以被 `import` 的包放哪里

这两个位置经常不是同一个地方。

## 1.1 `pip` 是什么、管理什么

`pip` 是 Python 官方生态里最基础、最通用的包安装工具。它主要负责把 PyPI 上的包安装到当前 Python 解释器对应的环境里。

### 概念

`pip` 主要管理的是：

- Python 包
- 依赖版本解析
- 从 PyPI 下载包并安装到当前环境

`pip` 不直接负责这些事情：

- 自动创建虚拟环境
- 自动给每个项目做依赖隔离
- 标准化管理整个项目的依赖声明和锁文件
- 管理 Python 解释器本身

所以 `pip` 更像“安装器”，而不是完整的项目管理器。

### 工作机制

执行下面这类命令时：

```bash
python -m pip install requests
python -m pip install "fastapi>=0.110"
python -m pip install -r requirements.txt
```

`pip` 会大致完成这些步骤：

1. 读取你指定的包和版本约束。
2. 解析依赖关系。
3. 下载 wheel 或源码包。
4. 构建并安装到当前 Python 环境的 `site-packages`。

### mac 下的路径理解

使用 `pip` 时要分两种位置：

1. 下载缓存放哪里  
2. 实际包装到哪里

在 macOS 下，`pip` 默认缓存常见位置是：

```bash
~/Library/Caches/pip
```

但真正能被 `import` 的包，不一定在这里，而是装进当前 Python 环境对应的 `site-packages`。

如果当前不在虚拟环境里，直接执行：

```bash
pip install xxx
```

那么包装到“当前 Python”的 `site-packages` 里，而不是装进某个项目目录。

### 隔离方式

`pip` 自己不做项目隔离。  
如果两个项目共用同一个 Python 环境，那么它们装的包就会互相影响。

所以 `pip` 最稳妥的常见用法是配合 `venv`：

```bash
python -m venv .venv
source .venv/bin/activate
python -m pip install fastapi
```

### 常用命令

```bash
python -m pip install requests
python -m pip install -r requirements.txt
python -m pip uninstall requests
python -m pip list
python -m pip show requests
python -m pip freeze
python -m pip cache dir
```

### 适用场景

`pip` 适合：

- 简单脚本或小项目
- 教程示例
- 需要最大兼容性的环境
- 已经明确知道当前 Python 环境来源时的直接安装

如果是认真维护的项目，通常建议和 `venv`、`uv` 或 `poetry` 搭配使用，而不是长期裸用系统环境。

## 1.2 `python -m venv .venv` 的作用与目录结构

`venv` 是 Python 标准库自带的虚拟环境方案，用来给项目创建独立的 Python 运行环境。

### 概念

这条命令：

```bash
python -m venv .venv
```

含义是：在当前目录下创建一个叫 `.venv` 的虚拟环境。

这个目录里会包含：

- 一套独立的 Python 可执行文件
- 一套独立的 `pip`
- 一套独立的 `site-packages`

### 工作机制

创建完成后，项目目录通常会多出：

```text
项目目录/
├── .venv/
│   ├── bin/
│   ├── lib/python3.x/site-packages/
│   └── pyvenv.cfg
```

激活环境后：

```bash
source .venv/bin/activate
```

后续执行的 `python` 和 `pip` 就优先指向 `.venv` 里的那一套。

### mac 下的路径理解

当你在项目根目录执行：

```bash
python -m venv .venv
```

虚拟环境就创建在当前项目下的：

```bash
./.venv
```

如果之后安装依赖：

```bash
python -m pip install fastapi
```

那么包会进入：

```bash
.venv/lib/pythonX.Y/site-packages
```

这和“直接装到当前 Python 的全局 `site-packages`”是两回事。

### 隔离方式

`venv` 的隔离本质是：

- 项目 A 有自己的 `.venv`
- 项目 B 有自己的 `.venv`
- 两边的 `site-packages` 分开

这样即使项目 A 用 `fastapi==0.110`，项目 B 用 `fastapi==0.115`，也可以互不影响。

### 常用命令

```bash
python -m venv .venv
source .venv/bin/activate
deactivate
python -m pip install -U pip
python -m pip install -r requirements.txt
```

### 适用场景

`venv` 适合：

- 希望使用最原生、最轻量的 Python 隔离方式
- 项目不需要额外的项目管理器
- 想和 `pip` 组合成传统但稳定的工作流

## 1.3 `uv` 如何管理依赖、缓存、Python 版本、项目隔离

`uv` 是新一代 Python 包与项目管理工具，它比 `pip` 更完整，既能管依赖，也能管虚拟环境、锁文件、Python 版本和命令运行。

### 概念

`uv` 的定位不是“每个项目一套 uv”，而是：

- 机器上通常只有一个全局 `uv`
- `uv` 可以管理多个 Python 版本
- `uv` 可以管理多个项目
- 每个项目再拥有自己的独立 `.venv`

也就是说，`uv` 是全局工具；项目隔离靠的不是“多个 uv”，而是“多个项目环境”。

### 工作机制

常见项目流转方式：

```bash
uv init myproj
cd myproj
uv add fastapi
uv add pytest --dev
uv run python main.py
uv sync
```

`uv` 会维护几个核心部分：

- `pyproject.toml`：项目依赖声明
- `uv.lock`：精确锁定版本
- `.venv`：当前项目自己的运行环境

`uv run` 会优先寻找当前项目对应的环境，而不是随便调用系统 Python。

### mac 下的路径理解

在 `uv` 的语境里，要分清楚三层位置：

1. `uv` 工具本身  
2. `uv` 全局管理的 Python  
3. 项目自己的 `.venv`

其中：

- 全局 `uv` 是一份工具，可供所有项目共用
- 全局 Python 会被 `uv` 统一保存和复用
- 每个项目自己的依赖仍然装在各自的 `.venv` 里

此外，`uv` 还有全局缓存目录，负责缓存下载结果、构建结果等内容。

### 隔离方式

`uv` 的项目隔离依赖于：

- 每个项目独立的 `.venv`
- 每个项目自己的 `pyproject.toml`
- 每个项目自己的 `uv.lock`

所以项目 A 和项目 B 即使都由同一个 `uv` 管理，依赖也不会混在一起。

可以把 `uv` 的职责拆成三层来记：

- 全局工具层：`uv`
- 全局资源层：`uv` 管理的 Python、缓存
- 项目隔离层：每个项目自己的 `.venv`

### 常用命令

```bash
uv init
uv add fastapi
uv add pytest --dev
uv remove fastapi
uv sync
uv run python main.py
uv venv
uv cache dir
uv python dir
uv python list --only-installed
```

### 适用场景

`uv` 适合：

- 新的 Python 项目
- 希望统一管理依赖、锁文件、虚拟环境、Python 版本
- 想保留接近 `pip` 的使用习惯，但需要更现代的工作流
- 希望项目目录结构清晰，并默认使用项目内 `.venv`

## 1.4 `poetry` 如何管理依赖与虚拟环境

`poetry` 是以“项目”为中心的 Python 依赖管理和打包工具。

### 概念

`poetry` 主要管理：

- `pyproject.toml`
- `poetry.lock`
- 项目虚拟环境
- 构建与发布流程

相比 `pip`，`poetry` 更强调“先声明项目依赖，再统一解析、锁定、安装”。

### 工作机制

常见流程如下：

```bash
poetry new myproj
cd myproj
poetry add fastapi
poetry add pytest --group dev
poetry install
poetry run python main.py
```

这里：

- 依赖声明写入 `pyproject.toml`
- 精确解析结果写入 `poetry.lock`
- 安装发生在 Poetry 管理的虚拟环境里

### mac 下的路径理解

`poetry` 默认会把虚拟环境放在自己的缓存目录下，而不是直接放进项目目录。

常见默认位置是：

```bash
~/Library/Caches/pypoetry/virtualenvs
```

也可以改成项目内 `.venv`，这样每个项目的环境就和代码放在一起，更直观。

### 隔离方式

`poetry` 的隔离本质上也是“每个项目一个虚拟环境”。

差别只在于默认环境位置：

- 默认：集中放在 Poetry 缓存目录下
- 可配置：放到项目根目录的 `.venv`

推荐在长期维护项目中开启项目内虚拟环境：

```bash
poetry config virtualenvs.in-project true
```

这样项目目录下就会出现：

```text
项目目录/
├── pyproject.toml
├── poetry.lock
└── .venv/
```

### 常用命令

```bash
poetry add fastapi
poetry add pytest --group dev
poetry remove fastapi
poetry install
poetry update
poetry run python main.py
poetry env info --path
poetry env list --full-path
poetry config --list
```

### 适用场景

`poetry` 适合：

- 团队协作项目
- 需要稳定依赖锁定
- 需要构建和发布 Python 包
- 希望围绕 `pyproject.toml` 做规范化管理

## 1.5 `conda` 的定位与和前面三者的区别

`conda` 和 `pip`、`uv`、`poetry` 的差别，核心不在命令写法，而在管理层级。

### 概念

`conda` 不只是 Python 包管理器，它更像：

- 环境管理器
- Python 版本管理器
- 二进制依赖管理器

它不只会装 Python 包，还经常负责安装底层依赖，例如科学计算、机器学习、CUDA 相关组件。

### 工作机制

常见用法：

```bash
conda create -n myenv python=3.11
conda activate myenv
conda install numpy pandas
```

`conda` 会创建一个独立环境，并把指定 Python 版本和相关依赖一起装进去。

### mac 下的路径理解

`conda` 环境通常放在 Conda 自己管理的环境目录下。  
如果使用的是 Anaconda，常见形式类似：

```bash
/opt/anaconda3/envs/环境名
```

这意味着 Conda 更像是“整套运行时环境”的隔离，而不是只关心 `site-packages`。

### 隔离方式

`conda` 的隔离方式比 `venv` 更完整：

- 独立环境目录
- 独立 Python 版本
- 独立包集合
- 独立二进制依赖

所以它特别适合依赖复杂、底层库较重的场景。

### 常用命令

```bash
conda create -n myenv python=3.11
conda activate myenv
conda install numpy pandas
conda list
conda env list
conda remove -n myenv --all
```

### 适用场景

`conda` 适合：

- 数据科学
- 深度学习
- 需要 CUDA 或复杂二进制依赖
- 需要同时管理 Python 解释器和底层依赖

和前面三者相比，可以这样粗略区分：

- `pip`：最基础的包安装器
- `uv`：更现代的一体化项目管理工具
- `poetry`：偏项目依赖管理和打包发布
- `conda`：偏完整运行环境和底层依赖管理

## 1.6 4 个工具的选型建议

选择时，不要只看“哪个命令更好记”，而要先看自己要管理的对象到底是什么。

### 通用选择思路

- 只是快速装包，且能明确当前 Python 来源：`pip`
- 想要传统稳定工作流：`pip + venv`
- 新项目，希望依赖、虚拟环境、锁文件、Python 版本统一管理：`uv`
- 团队项目，尤其需要打包和发布：`poetry`
- 科学计算、深度学习、CUDA、复杂二进制依赖：`conda`

### 简单对比

| 工具 | 主要管理对象 | 默认是否强调项目隔离 | 是否适合管理 Python 版本 | 是否擅长二进制依赖 |
| --- | --- | --- | --- | --- |
| `pip` | 当前环境中的 Python 包 | 否 | 否 | 否 |
| `venv` | 项目级虚拟环境 | 是 | 否 | 否 |
| `uv` | 项目、依赖、环境、锁文件、Python 版本 | 是 | 是 | 否 |
| `poetry` | 项目依赖、锁文件、虚拟环境、打包 | 是 | 否 | 否 |
| `conda` | 环境、Python 版本、包、底层依赖 | 是 | 是 | 是 |

### 推荐实践

如果是普通 Python 新项目，优先考虑 `uv`。  
如果是传统兼容项目，可以稳定使用 `pip + venv`。  
如果项目需要打包发布，`poetry` 依然是很常见的选择。  
如果是科学计算、CUDA、底层依赖复杂的环境，优先使用 `conda`。

## 1.7 当前这台 Mac 的实际路径与查看命令

这一节只记录当前机器已经确认过的实际路径，用于把上面的概念和本机环境对应起来。

### 当前已确认的实际路径

- 默认 `python` / `python3` 当前指向：

```bash
/opt/anaconda3/bin/python3
```

- 默认 `pip` / `pip3` 当前指向：

```bash
/opt/anaconda3/bin/pip
```

- 当前不在虚拟环境里时，实际 `site-packages` 在：

```bash
/opt/anaconda3/lib/python3.12/site-packages
```

- 当前用户级 `site-packages` 在：

```bash
/Users/sdybdc/.local/lib/python3.12/site-packages
```

- `uv` 可执行文件在：

```bash
/Users/sdybdc/.local/bin/uv
```

- `uv` 全局 Python 目录在：

```bash
/Users/sdybdc/.local/share/uv/python
```

- `uv` 缓存目录在：

```bash
/Users/sdybdc/.cache/uv
```

### 当前环境的理解

这台机器当前默认使用的是 Anaconda 的 Python，而不是系统自带 Python，也不是某个项目里的 `.venv`。

因此，如果直接执行：

```bash
pip install xxx
```

包默认会进入：

```bash
/opt/anaconda3/lib/python3.12/site-packages
```

也就是说，这样的安装是“装到当前 Conda 基础环境”，而不是“装到某个项目自己的虚拟环境”。

### 本机查看命令

下面这些命令可以直接用来确认当前机器的环境位置：

```bash
which -a python python3 pip pip3 uv
python3 -c "import sys,sysconfig,site; print(sys.executable); print(sysconfig.get_paths()['purelib']); print(getattr(site, 'USER_SITE', None))"
python3 -m pip show pip
python3 -m pip cache dir
uv python dir
uv cache dir
uv python list --only-installed
poetry env info --path
poetry env list --full-path
```

### 本章结论

后续实际开发时，优先避免把业务项目依赖直接装进当前默认环境。  
更稳妥的方式是：

- 传统工作流：项目内使用 `.venv`
- 新项目工作流：使用 `uv` 管理项目依赖和环境
- 需要发布或团队协作较强时：使用 `poetry`
- 需要复杂运行时依赖时：使用 `conda`
