# PaperChatAgent V1 需求文档

## 1. 项目概述

### 1.1 项目背景

科研人员和学生在论文调研过程中，通常需要经历研究方向澄清、论文检索、资料收集、内容阅读、观点比对和研究结论沉淀等多个环节。现有工具往往只能覆盖单点能力，例如检索、翻译、摘要或问答，缺少围绕研究主题持续推进的统一工作台。

PaperChatAgent 的目标是构建一个面向科研人员和学生的智能论文调研工具，以对话为入口，以研究工作区为沉淀对象，以研究任务为执行单元，帮助用户围绕某个研究主题持续开展探索、整理语料并进行基于引用依据的问答。

### 1.2 问题定义

当前论文调研常见痛点包括：

- 用户在开始调研时往往无法准确表达研究问题，需要与系统反复讨论才能明确研究范围。
- 检索、上传资料、阅读理解、记录结论和后续追问通常分散在多个工具中，流程割裂。
- 研究过程中形成的上下文难以长期保存，历史会话与资料之间缺少稳定关联。
- 论文问答结果缺少明确来源，难以在研究和写作场景中直接使用。

### 1.3 产品目标

PaperChatAgent V1 的产品目标是：

- 提供一个以聊天为默认入口的论文调研工作台。
- 支持用户通过对话澄清研究需求，并在过程中上传论文或其他研究资料。
- 在方向相对明确后，由 AI 建议研究任务，用户确认后创建研究工作区并启动后台研究流程。
- 将外部检索与用户上传资料沉淀为可持续复用的主题级语料上下文。
- 让用户在后续问答中获得附带引用依据的回答，而不是仅得到无出处的总结。

### 1.4 V1 定位

PaperChatAgent V1 是一个主题级语料库上的论文问答助手，不是正式综述报告生成器。V1 首要解决的是“研究方向澄清、主题探索、资料沉淀、引用式问答”这一主链路，而不是自动产出完整调研报告。

## 2. 目标用户与场景

### 2.1 目标用户

V1 核心用户为研究生和博士生，也可覆盖有论文调研需求的科研工程师与高年级本科生。

### 2.2 核心使用场景

- 选题探索：用户尚未完全确定研究方向，希望与 AI 讨论、缩小范围并形成研究主题。
- 主题调研：用户围绕一个研究主题收集论文、对比观点、梳理脉络。
- 论文理解：用户上传单篇或多篇论文，希望系统辅助阅读、总结和回答问题。
- 持续追问：用户在已有研究语料基础上进行多轮追问，而不是每次从零开始。

### 2.3 用户价值

- 降低研究启动成本，让用户从“我还没想清楚”顺滑过渡到“我有一个可研究的主题工作区”。
- 提高资料组织效率，把聊天、论文、任务和研究结果统一在一个工作台内。
- 提升研究问答可信度，让回答能够关联到论文来源和引用依据。

## 3. 产品信息架构

### 3.1 总体结构

V1 采用登录后进入工作台的产品结构，首页默认进入聊天页。整体界面包含以下部分：

- 顶层入口：登录、注册。
- 主工作台：首页默认聊天页。
- 左侧边栏上半区：功能导航。
- 左侧边栏下半区：历史会话记录，按工作区分组展示，交互风格参考 Codex。

### 3.2 功能导航

侧边栏上半区包含以下模块：

- 聊天
- 知识库
- MCP 服务
- Skills
- 智能体
- 后台任务
- 数据看板

其中 V1 真正进入主链路的模块为：

- 聊天
- 知识库
- 智能体
- 后台任务
- 数据看板

其中 V1 仅做配置预留的模块为：

- MCP 服务
- Skills

### 3.3 核心对象定义

为避免产品语义混乱，V1 文档统一使用以下对象定义：

- `用户`：使用 PaperChatAgent 的登录用户。
- `默认收件箱会话`：用户首次进入系统后的默认聊天会话，用于需求澄清和研究方向讨论，在创建工作区前先承载早期对话。
- `研究工作区`：围绕某一研究主题建立的长期空间，用于组织会话、研究任务、知识引用和结果沉淀。
- `研究任务`：用户确认后启动的异步研究流程，用于检索、解析、建索引、主题探索和结果生成。
- `全局知识库`：当前账号内全局复用的论文和资料集合，可跨工作区复用。
- `工作区私有知识库`：仅归属于特定研究工作区的补充资料集合。
- `主题探索包`：研究任务完成后形成的主题语料上下文，包括论文集合、结构化分析结果与可继续问答的研究背景。
- `预置研究智能体`：系统内置的研究流程智能体及其编排视图，用户主要负责选择、观察和使用。
- `引用依据`：回答所引用的论文来源信息，默认以论文级展示，在条件满足时可展开到更细粒度。

### 3.4 核心对象关系

- 默认聊天先存在于 `默认收件箱会话` 中。
- 当研究方向明确后，`默认收件箱会话` 可转化为 `研究工作区` 的上下文来源之一。
- 知识库采用“当前账号内全局公共库 + 工作区私有补充库”的双层结构。
- `研究任务` 完成后产出 `主题探索包`，并回流到聊天页供用户继续问答。

## 4. 核心业务流程

V1 核心业务流程为：

`登录 -> 默认聊天 -> 澄清需求/上传资料 -> AI 建议研究任务 -> 用户确认 -> 创建工作区 -> 后台执行 -> 结果回流问答`

### 4.1 详细流程说明

1. 用户完成注册或登录后进入默认聊天页。
2. 系统创建或打开该用户的 `默认收件箱会话`。
3. 用户通过对话描述研究问题，也可以上传论文 PDF 或其他研究资料。
4. AI 根据聊天内容帮助用户明确研究方向、关键词、问题边界和资料范围。
5. 当系统判断方向已具备可执行条件时，AI 提出研究任务建议。
6. 用户打开轻量确认面板，确认或修改研究主题、关键词、时间范围、来源和上传资料。
7. 用户确认启动后，系统创建 `研究工作区`，并生成对应的 `研究任务`。
8. 后台任务异步执行，完成论文检索、资料解析、知识入库和主题探索。
9. 系统将结果沉淀为 `主题探索包`，并在聊天中回流可继续问答的研究上下文。
10. 用户可在工作区中继续追问、查看任务进度、浏览资料和切换历史会话。

## 5. 功能需求

### 5.1 登录注册

#### 用户价值

为用户提供基础身份能力，使其能够持久保存研究会话、知识资料和工作区。

#### 页面职责

- 提供注册页面和登录页面。
- 支持邮箱密码登录注册。

#### 核心行为

- 用户可使用邮箱和密码注册账号。
- 用户可使用邮箱和密码登录。
- 登录成功后进入主工作台默认聊天页。

#### 边界限制

- V1 不要求第三方登录。
- V1 不要求复杂角色权限体系。
- 所有登录用户在功能权限上保持一致。

### 5.2 聊天首页与默认收件箱会话

#### 用户价值

让用户无需先创建项目或工作区，就能直接与 AI 讨论研究需求。

#### 页面职责

- 作为登录后的默认首页。
- 承载需求澄清、资料上传、研究方向对齐和任务建议。

#### 核心行为

- 系统默认打开 `默认收件箱会话`。
- 用户可以围绕研究方向发起多轮对话。
- 用户可以在聊天中上传 PDF 或相关研究资料。
- AI 可以基于对话内容建议下一步研究任务。
- 在工作区尚未创建前，聊天记录保留在默认收件箱会话中。

#### 边界限制

- 聊天首页不是最终报告页。
- 未确认研究任务前，不自动生成正式工作区。

### 5.3 研究工作区

#### 用户价值

为用户提供围绕研究主题长期积累资料和结论的稳定空间。

#### 页面职责

- 按研究主题组织会话、任务、知识和结果。
- 承接从默认收件箱会话转化而来的研究上下文。

#### 核心行为

- 当用户确认研究任务后，系统自动创建工作区。
- 工作区下挂该主题相关的历史会话、研究任务和结果资产。
- 历史记录按工作区分组展示，支持快速回到某个主题继续研究。
- 工作区支持只读分享链接，便于对外查看结果。

#### 边界限制

- V1 不支持多人协作编辑。
- V1 不支持复杂的团队权限控制。

### 5.4 知识库

#### 用户价值

帮助用户统一管理论文和研究资料，并将其挂接到研究过程。

#### 页面职责

- 管理账号内全局知识库和工作区私有知识库。
- 提供资料上传、资料查看、资料挂接和复用能力。

#### 核心行为

- 用户可以上传 PDF 到系统。
- 系统支持通过 arXiv 检索引入论文。
- 账号内全局知识库可跨工作区复用。
- 工作区私有知识库可补充仅属于当前主题的资料。
- 用户可将资料挂接到指定工作区或研究任务。

#### 边界限制

- V1 不做重度人工标注编辑器。
- V1 不以知识库页面作为研究入口，研究入口仍然是聊天页。

### 5.5 智能体页面

#### 用户价值

让用户看到系统如何围绕研究任务进行多阶段执行，并理解当前工作流状态。

#### 页面职责

- 展示预置研究智能体和研究流程编排视图。
- 显示关键节点的职责、顺序和状态。

#### 核心行为

- 页面以 Paper-Agent 六大节点作为研究工作流编排蓝图。
- 用户可以查看各节点的职责说明和执行状态。
- 用户主要使用系统预置研究智能体，而不是在 V1 中自由创建复杂智能体。

#### 边界限制

- V1 不以智能体页面为主要配置中心。
- V1 不承诺正式调研报告生成功能。
- 与正式报告相关的节点仅作为流程蓝图和后续扩展方向展示，不构成 V1 对外交付物。

### 5.6 后台任务

#### 用户价值

将耗时的研究流程异步化，避免阻塞聊天体验。

#### 页面职责

- 展示研究任务的状态、进度和结果摘要。
- 作为确认研究方向后执行工作流的承载模块。

#### 核心行为

- AI 建议研究任务后，用户可通过轻量确认面板启动任务。
- 任务异步执行，包括检索、下载、解析、建索引和主题探索等步骤。
- 用户可以查看任务的排队、执行中、完成、失败等状态。
- 任务完成后，结果回流到工作区和聊天页。

#### 边界限制

- V1 后台任务重点承接研究任务，不强调定时订阅和运维任务。
- V1 不要求复杂任务编排可视化编辑能力。

### 5.7 数据看板

#### 用户价值

让用户在主工作台中查看系统层面的运营和运行概况。

#### 页面职责

- 作为主侧边栏中的一个可见模块，展示系统运营指标。

#### 核心行为

- 展示系统级别的指标信息，例如任务量、处理状态分布、活跃情况或其他平台指标。
- 与主工作台并列存在，作为可查看模块而非后台管理区。

#### 边界限制

- V1 不额外拆分独立后台管理区。
- V1 看板以查看为主，不承载复杂运营管理能力。

### 5.8 MCP 服务 / Skills 预留

#### 用户价值

为后续扩展工具接入和技能增强预留产品位置。

#### 页面职责

- 在导航中提供 MCP 服务和 Skills 的入口。
- 为后续接入第三方能力保留配置位。

#### 核心行为

- V1 提供基础页面或配置占位。
- 不进入核心研究链路，不作为 V1 主交付能力。

#### 边界限制

- V1 不要求用户真正完成 MCP 接入。
- V1 不要求 Skills 参与研究任务执行。

## 6. 数据与系统约束

### 6.1 技术基线

V1 默认技术基线如下：

- 前端：Vue 3 + Vite
- 工作流编排：LangGraph
- 关系数据：MySQL
- 向量检索：ChromaDB
- 异步任务：Redis 队列 + Worker
- 对象存储：MinIO

### 6.2 系统设计约束

- 系统采用多阶段研究流程，但 V1 以主题探索和问答闭环为主。
- LangGraph 作为主编排框架，负责组织预置研究智能体的执行流程。
- MinIO 作为统一对象存储层，用于保存原始论文、上传文件和可持久化工件。
- MySQL 用于保存用户、会话、工作区、任务和知识关联等结构化数据。
- ChromaDB 用于主题语料的向量检索。
- Redis Worker 用于执行长耗时研究任务。

### 6.3 资料来源约束

- V1 资料入口仅覆盖 arXiv 和用户上传 PDF。
- 论文处理深度采用“摘要优先、可获取全文则继续解析”的策略。

### 6.4 引用机制约束

- 所有研究问答结果必须附带 `引用依据`。
- 默认以论文级展示来源信息。
- 当解析结果足以支持更细粒度展示时，可进一步展示段落或片段级依据。

## 7. 非功能需求

### 7.1 性能与响应

- 默认聊天交互应保持可连续使用，不因长任务而完全阻塞。
- 研究任务应通过异步机制运行，并可查询进度状态。

### 7.2 可追溯性

- 每个研究结果应能够回溯到对应论文或资料来源。
- 用户应能知道某个主题探索包来源于哪些任务和哪些资料。

### 7.3 可用性

- 用户进入系统后应直接进入聊天，而不需要先理解复杂配置。
- 从聊天到工作区的过渡应尽量自然，不要求用户提前掌握产品结构。

### 7.4 权限边界

- V1 默认所有登录用户功能权限一致。
- 工作区和私有资料默认仅自己可见。
- 对外分享仅支持只读结果访问。

## 8. V1 非目标

以下内容明确不属于 V1 必做范围：

- 第三方平台集成，例如飞书、Slack、企业微信等。
- 正式调研报告自动生成与导出。
- 多人协作编辑工作区。
- MCP 服务和 Skills 深度接入主研究链路。
- 复杂管理员后台和多角色权限体系。
- 高级人工标注、片段精修和知识库深度运营工具。

## 9. 验收标准

### 9.1 登录注册

- 用户可以通过邮箱密码完成注册和登录。
- 登录成功后默认进入聊天页。

### 9.2 默认聊天页

- 系统自动生成或打开默认收件箱会话。
- 用户可以在聊天中上传 PDF 并继续对话。
- AI 能在方向相对明确后提出研究任务建议。

### 9.3 工作区

- 用户确认研究任务后，系统能自动创建研究工作区。
- 历史会话能够按工作区分组展示。
- 工作区支持只读分享链接。

### 9.4 知识库

- 用户可以管理账号内全局知识库与工作区私有知识库。
- 资料可从 arXiv 检索引入或由用户上传。
- 知识资料能正确挂接到工作区或研究任务。

### 9.5 智能体

- 页面能展示 Paper-Agent 六大节点研究流程的编排视图。
- 用户能看到节点职责和状态信息。

### 9.6 后台任务

- 用户可在轻量确认面板后启动研究任务。
- 任务状态至少包含待执行、执行中、已完成、失败。
- 任务结果能回流到聊天与工作区。

### 9.7 数据看板

- 数据看板作为主侧边栏模块可访问。
- 看板可展示基础系统运营指标。

### 9.8 MCP 服务 / Skills

- 两个模块在导航中存在。
- 页面具备配置预留或占位能力，但不阻塞核心研究流程。

### 9.9 问答与引用

- 研究任务完成后，系统形成可继续问答的主题探索包。
- 后续问答必须带有引用依据。

## 10. 后续规划

V1 之后可考虑的方向包括：

- 调研报告生成
- 第三方平台集成
- MCP / Skills 深度接入
- 协作能力增强
