# PaperChatAgent V1 需求文档

## 1. 项目概述

### 1.1 项目背景

研究者在做论文调研时，往往需要在“澄清研究目标、收集资料、理解文献、持续追问、沉淀结论”之间来回切换。多数工具只能覆盖搜索、摘要、问答、翻译中的某一段，缺少一个以聊天为主链路、又能把资料、工具和异步研究流程自然接起来的统一产品。

PaperChatAgent V1 的目标是做一个聊天优先的研究助手：用户从聊天开始，逐步明确研究方向，上传或沉淀资料，必要时调用智能体工作流、外部工具和知识检索能力，并通过后台任务观察异步研究流程的进展。

### 1.2 问题定义

当前论文调研常见痛点包括：

- 用户一开始往往说不清研究问题，需要多轮讨论才能收束目标。
- 文档、搜索结果、问答过程和结论沉淀分散在多个工具中。
- 历史会话和资料缺少稳定关联，后续追问经常从零开始。
- 模型回答无法稳定说明“为什么这么回答”，缺少引用来源或检索依据。
- 复杂研究流程缺少可观察的执行界面，用户不知道系统当前做到哪一步。

### 1.3 产品目标

PaperChatAgent V1 的目标是：

- 提供一个类似 GPT 的聊天主链路研究体验。
- 让用户通过聊天澄清研究主题、上传文档、持续追问。
- 让知识库作为独立模块管理资料解析、切片、向量化与检索来源。
- 让智能体作为独立模块提供可扩展的研究工作流能力。
- 让模型在聊天阶段自主判断是否需要检索知识库、调用智能体、MCP 或 Skills。
- 在用户确认主题后，通过后台任务展示智能体工作流的异步执行进度。

### 1.4 V1 定位

PaperChatAgent V1 是一个聊天优先的研究助手，不是正式调研报告平台，也不是一个以知识库管理为主的资料运营系统。

V1 关注的是这条主链路：

`聊天澄清 -> 资料沉淀 -> 模型自主检索 / 调用工具 -> 异步研究执行 -> 结果回流聊天`

## 2. 目标用户与场景

### 2.1 目标用户

V1 核心面向：

- 研究生
- 博士生
- 需要进行论文调研的科研工程师
- 需要完成课程研究或毕业论文的高年级本科生

### 2.2 核心使用场景

- `选题探索`：用户尚未完全确定研究方向，希望通过聊天逐步澄清问题边界。
- `论文理解`：用户上传一篇或多篇论文，希望系统辅助阅读、总结和继续追问。
- `主题调研`：用户围绕一个主题持续积累资料，并在已有语料上反复提问。
- `流程执行`：用户确认主题后，希望系统异步完成检索、解析、建库、主题探索等步骤，并能看到进度。

### 2.3 用户价值

- 降低研究启动成本，让用户从模糊想法自然走到可执行主题。
- 提高资料管理效率，把聊天、资料、工具和任务组织在同一产品内。
- 提高回答可信度，让模型能在需要时基于知识库或工具结果作答。
- 提高过程可见性，让用户能在后台任务中看到智能体工作流的执行状态。

## 3. 产品信息架构

### 3.1 总体结构

V1 采用登录后进入聊天页的产品结构。聊天是主入口，其余模块作为能力层、配置层或观测层存在。

主界面包含：

- 顶层入口：登录、注册
- 主内容区：聊天页
- 左侧导航：模块导航
- 左侧历史区：最近会话列表

### 3.2 模块导航

V1 导航统一为以下模块：

- 聊天
- 知识库
- 智能体
- MCP 服务
- Skills
- 模型
- 后台任务
- 数据看板

各模块定位如下：

- `聊天`：主链路入口
- `知识库`：资料解析与检索来源管理
- `智能体`：内置工作流与后续扩展能力
- `MCP 服务`：外部工具配置
- `Skills`：技能配置
- `模型`：模型配置与模型路由
- `后台任务`：智能体 / 工作流执行进度展示
- `数据看板`：系统观测与指标展示

### 3.3 核心对象定义

为避免语义混乱，V1 文档统一使用以下对象：

- `用户`：登录到 PaperChatAgent 的使用者。
- `会话`：用户长期使用和切换的主对象，承载聊天历史与后续追问。
- `消息`：会话中的单条用户消息、AI 消息、系统消息或结构化事件消息。
- `知识库`：一个资料容器，包含主题相关文件、解析结果和向量检索来源。
- `知识文件`：知识库中的单个文件、论文或导入资料。
- `研究任务`：用户确认主题后触发的异步研究执行对象。
- `工作流运行`：一次智能体 / 工作流的真实运行实例。
- `节点运行`：工作流中单个节点的状态与进度记录。
- `智能体工作流`：当前系统内置或后续扩展的流程定义。
- `MCP 服务配置`：当前接入的 MCP 服务定义。
- `Skill 配置`：当前接入的 Skill 定义。
- `模型路由配置`：不同模型槽位及供应商配置。
- `看板指标`：系统观测指标，例如模型调用量、输入输出量、任务状态分布。

### 3.4 核心对象关系

- 聊天围绕 `会话` 展开。
- `知识库` 是独立模块，但可在聊天中被模型自主检索。
- 用户可在聊天中上传资料，并绑定到已有知识库或新建知识库。
- 当主题清晰后，聊天可触发 `研究任务`。
- `研究任务` 会创建一条 `工作流运行` 记录，并在后台任务页展示节点进度。
- 智能体、MCP、Skills 都可以在聊天阶段作为被调用能力出现。

## 4. 核心业务流程

V1 主业务流程为：

`登录 -> 聊天 -> 澄清主题 / 上传资料 -> 绑定或创建知识库 -> 模型自主判断是否检索或调用工具 -> 用户确认主题 -> 触发后台任务 -> 后台任务页展示智能体进度 -> 结果回流聊天`

### 4.1 详细流程说明

1. 用户登录后进入聊天页。
2. 系统默认打开最近会话或创建首个默认会话。
3. 用户通过聊天描述研究目标，也可以上传 PDF 或其他文档。
4. 系统可引导用户澄清研究主题、关键词、问题边界和预期产出。
5. 若用户上传资料，系统允许其绑定到已有知识库或创建新知识库；知识库名称可由用户填写，也可由 AI 建议。
6. 在聊天过程中，模型可自主判断是否需要检索知识库、是否需要调用智能体、MCP 或 Skills。
7. 当主题确认到可以执行研究任务时，系统触发研究任务确认。
8. 后台任务异步执行，展示智能体 / 工作流节点进度。
9. 研究结果、报告摘要或关键发现回流到聊天中，支持继续追问。

## 5. 功能需求

### 5.1 登录注册

#### 用户价值

为用户提供稳定的身份体系，保存会话、资料、配置和任务记录。

#### 页面职责

- 提供注册页和登录页
- 支持邮箱密码登录注册

#### 核心行为

- 用户可以使用邮箱密码注册
- 用户可以使用邮箱密码登录
- 登录成功后默认进入聊天页

#### 边界限制

- V1 不要求第三方登录
- V1 不要求复杂角色权限体系

### 5.2 聊天

#### 用户价值

让用户以最低门槛开始研究，不需要先理解系统结构或先配置模块。

#### 页面职责

- 作为登录后的默认首页
- 承载研究主题澄清、多轮追问、资料上传与任务触发
- 通过动态提示区展示当前会话判断、专业步骤与可执行研究草案

#### 核心行为

- 系统默认打开最近会话或创建新会话
- 用户首次进入时既可以普通交流，也可以逐步转入深入研究
- 用户可持续与 AI 对话，逐步明确研究主题
- AI 会根据当前聊天内容动态判断是“普通交流”、“方向探索”还是“可深入研究”
- 当信息逐渐充分时，聊天侧栏会实时整理下一步建议、待补充信息与研究草案
- 用户可在聊天中上传 PDF 或补充研究资料
- 模型可基于上下文自主决定是否检索知识库
- 模型可按需调用智能体、MCP 或 Skills
- 当主题足够明确时，聊天可触发研究任务
- 研究任务完成后，结果回流聊天供用户继续追问

#### 边界限制

- 聊天页是主链路入口，但不承担复杂配置管理
- 聊天页不直接展示全量系统指标

### 5.3 知识库

#### 用户价值

帮助用户把上传文档、导入资料和后续解析结果沉淀为可被聊天使用的检索来源。

#### 页面职责

- 管理知识库列表页、知识库详情页中的知识文件、解析状态和索引状态
- 管理文档上传、资料导入和知识库命名

#### 核心行为

- 用户可手动创建知识库
- 知识库按名称列表展示，而不是按工作区或公私域分类展示
- 用户点击某个知识库后，可进入知识库详情页查看相关文件与解析状态
- 用户可上传 PDF 或其他资料到知识库
- 系统可在聊天过程中建议用户将资料绑定到某个知识库
- 系统也可在用户明确主题后，自动建议创建新的知识库
- 知识库中的资料会进入解析、切片、向量化等流程
- 聊天阶段模型可自主判断是否使用知识库检索结果

#### 边界限制

- 知识库不是 V1 的主入口
- V1 不做重度人工标注与复杂知识运营工具

### 5.4 智能体

#### 用户价值

让用户了解系统有哪些内置研究能力，以及这些能力如何在研究过程中被调用。

#### 页面职责

- 展示智能体列表页与智能体详情页
- 在详情页中展示节点结构、节点配置与执行链路

#### 核心行为

- 页面按列表展示内置智能体、自定义智能体和项目中的子 Agent
- 用户点击某个智能体后，可进入详情页查看节点结构
- 用户可在节点详情中配置模型槽位
- 用户可为节点选择通用模型、自定义智能体或项目中的子 Agent
- 上一个节点的输出可作为下一个节点的输入来源
- 聊天阶段可把该工作流作为工具能力调用
- 后续支持新增其他智能体或工作流

#### 边界限制

- V1 不支持用户自由编排复杂智能体流程
- 智能体页主要展示与配置能力，不承担系统总调度

### 5.5 MCP 服务

#### 用户价值

让用户能够查看和维护外部工具服务接入情况。

#### 页面职责

- 展示当前已配置 MCP 服务
- 提供导入本地 MCP 服务与新增服务配置的入口

#### 核心行为

- 页面按列表展示 MCP 服务
- 支持导入本地 MCP 服务
- 支持新增 MCP 服务配置

#### 边界限制

- V1 不要求 MCP 深度进入主研究链路
- V1 先以配置和展示能力为主

### 5.6 Skills

#### 用户价值

让用户能够查看和维护系统可用技能集合。

#### 页面职责

- 展示当前已配置 Skills
- 提供导入本地 Skills 与新增 Skill 的入口

#### 核心行为

- 页面按列表展示当前 Skills
- 支持导入本地 Skills
- 支持新增自定义 Skills

#### 边界限制

- V1 不要求 Skills 深度接入核心研究流程
- V1 不承诺提供复杂 Skill 编排体验

### 5.7 模型

#### 用户价值

让系统具备统一的模型配置与路由能力，支持不同阶段使用不同模型。

#### 页面职责

- 展示模型配置与逻辑模型槽位
- 展示模型路由关系

#### 核心行为

- 管理 `conversation`、`tool_call`、`reasoning`、`embedding`、`rerank` 等模型槽位
- 支持配置供应商、模型名、基础地址等信息
- 支持按逻辑能力路由到不同模型

#### 边界限制

- V1 不下沉到复杂的实验平台能力
- V1 不做多租户模型管理

### 5.8 后台任务

#### 用户价值

让用户在主题确认后看到智能体 / 工作流的真实执行进度，而不是只看到一个模糊的加载态。

#### 页面职责

- 展示研究任务列表
- 展示工作流运行状态、当前节点、节点进度和结果摘要

#### 核心行为

- 聊天确认主题后可创建研究任务
- 任务以异步方式执行
- 后台任务页展示智能体 / 工作流执行进度
- 节点状态至少包括待执行、运行中、已完成、失败
- 任务结果可回流聊天继续使用

#### 边界限制

- 后台任务页不是通用运维平台
- 页面重点是“展示智能体进度”，而不是调度各种无关任务

### 5.9 数据看板

#### 用户价值

让用户或系统维护者看到平台运行层面的核心指标。

#### 页面职责

- 展示系统观测指标

#### 核心行为

- 展示模型调用量
- 展示输入输出量
- 展示任务状态分布
- 展示其他基础运行指标

#### 边界限制

- V1 不做复杂 BI 或运营管理后台

## 6. 数据与系统约束

### 6.1 技术基线

V1 默认技术基线如下：

- 前端：Vue 3 + Vite
- 后端：FastAPI
- 工作流：LangGraph + AutoGen
- 关系数据库：MySQL
- 对象存储：MinIO
- 向量数据库：ChromaDB
- 异步执行：Python 进程内异步任务

### 6.2 系统设计约束

- 聊天是唯一主链路
- 知识库、智能体、MCP、Skills、模型、后台任务、看板都是独立模块
- 模型在聊天阶段自主判断是否需要检索知识库或调用外部能力
- 后台任务重点展示智能体工作流的执行进度
- 当前版本不引入独立 Worker + 队列，避免实现混乱
- 当前版本的异步任务必须具备完整恢复设计：
  - 自动重试
  - 节点暂停
  - 从失败节点续跑
  - 备用模型切换
  - 节点 checkpoint 持久化

### 6.3 引用与可追溯性约束

- 模型基于知识库或资料回答时，应尽可能返回引用依据
- 任务结果应可回溯到对应会话、资料或工作流运行

## 7. 非功能需求

### 7.1 响应体验

- 聊天应保持连续可用
- 长耗时步骤应通过后台任务异步执行

### 7.2 可追溯性

- 用户应能知道当前回答是否使用了知识库或工具调用
- 用户应能在后台任务页看到工作流执行到哪个节点

### 7.3 可扩展性

- 智能体模块应支持后续增加新的工作流
- MCP 服务与 Skills 模块应支持后续扩展更多配置项
- 模型模块应支持多供应商和多逻辑槽位

## 8. V1 非目标

以下内容不属于 V1 必做范围：

- 正式综述报告平台化交付
- 多人协作编辑
- 复杂权限体系
- 高级知识标注与运营工具
- 自由编排式智能体设计器
- 复杂 BI 后台

## 9. 验收标准

### 9.1 登录注册

- 用户可通过邮箱密码注册和登录
- 登录后默认进入聊天页

### 9.2 聊天

- 用户可创建和切换会话
- 用户可在聊天中上传资料并持续追问
- 模型可在聊天阶段基于上下文作答

### 9.3 知识库

- 用户可创建知识库并上传资料
- 资料具备解析与索引状态
- 聊天阶段可消费知识库内容

### 9.4 智能体

- 页面可展示当前内置工作流
- 用户可看到工作流节点定义与说明

### 9.5 MCP 服务 / Skills

- 页面可分别展示当前配置项
- 支持导入本地配置与新增配置入口

### 9.6 模型

- 页面或接口层能表达模型配置与模型路由关系

### 9.7 后台任务

- 主题确认后可触发研究任务
- 后台任务页可展示智能体 / 工作流执行进度
- 任务结果可回流聊天

### 9.8 数据看板

- 页面可展示模型调用、输入输出量、任务状态分布等基础指标

## 10. UI 落地映射

### 10.1 全局工作台布局

- 登录后统一采用 `左侧导航栏 + 右侧主视图` 的工作台结构
- 左侧导航统一包含：`聊天 / 知识库 / MCP 服务 / Skills / 智能体 / 模型 / 后台任务 / 数据看板`
- 聊天页、知识库页与智能体页保留最近会话区，支撑主链路中的上下文切换
- 智能体、模型、MCP、Skills、后台任务、数据看板页采用“主内容区 + 辅助信息卡”结构，突出配置、观测或流程说明
- 整体视觉保持浅色研究工作台风格，不做营销页或装饰性首页

### 10.2 页面级 UI 对应关系

- `登录页`：品牌说明卡 + 登录表单
- `注册页`：账号收益说明卡 + 注册表单
- `聊天页`：会话流、资料上传入口、动态研究提示区、底部输入区
- `知识库列表页`：按知识库名称展示列表，并提供新建 / 上传入口
- `知识库详情页`：查看某个知识库下的文件列表、解析状态与索引结果
- `MCP 服务页`：服务接入状态、服务列表、能力暴露说明与连接细节
- `Skills 页`：技能列表、启用状态、输入输出说明与使用场景
- `智能体列表页`：展示内置、自定义与项目子 Agent 列表
- `智能体详情页`：配置节点、模型槽位、执行器与上下游传递规则
- `模型页`：逻辑模型槽位、供应商配置、路由与降级策略
- `后台任务页`：任务状态统计、任务列表、节点进度与结果摘要
- `数据看板页`：模型调用指标、输入输出量、任务状态分布与近期事件

### 10.3 设计落地约束

- 侧边栏导航必须在所有工作台页面保持一致，避免模块入口缺失
- `模型` 作为独立模块必须进入导航，不再作为隐含配置存在
- `MCP 服务` 与 `Skills` 不再以“预留”文案展示，而应以可配置模块展示完整页面结构
- 知识库页与智能体页都应采用“列表页 + 详情页”的两层原型结构
- 聊天页右侧采用动态提示区，不再使用“默认收件箱会话 / 工作区创建”类旧概念
- 后台任务页与数据看板页需明确区分：前者关注单任务 / 单工作流进度，后者关注系统级指标
